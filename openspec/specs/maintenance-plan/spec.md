# 硫化机保养/精度计划排程规则

**状态：已实现** | **最近修订：2026-07-22** | **适用链路：基础数据、续作收尾、换活字块、新增排产、机台时间轴、产能扣减、结果校验、完成事件回填**

## Purpose

统一硫化机年度精准计划在 APS 中的触发、日期选择、长期在机强制下机、L/R 单控物理机同步、
不可生产时间轴、排程日志、班次原因和安排日期回填口径。保养固定在最终候选日 08:00 开始，
默认持续 7 小时；保养结束后继续预留默认 2.5 小时胶囊预热，最早 17:30 恢复生产。

本规则复用设备停机、清洗、班次管控和不可生产区间并集合并能力，不修改现有月计划、日计划、
胎胚库存、维修、清洗和换活字块的业务语义；仅明确正规换模与精度计划重叠时的并行时间轴。

## 数据口径

### Requirement: 年度精准计划必须包含可排程字段

年度精准计划 SHALL 至少包含以下字段：

| 字段 | 实体字段 | 业务含义 |
| --- | --- | --- |
| 硫化机台 | `machineCode` | 运行态机台编码；单控机台允许使用 L/R 侧编码 |
| 精度/保养类型 | `precisionType` | 保养类型 |
| 计划保养日期 | `planDate` | 来源计划日期 |
| 实际保养日期 | `actualDate` | MES/设备侧确认的实际完成日期 |
| 到期日期 | `dueDate` | 计划有效到期日 |
| 到期天数 | `daysToDue` | 距离到期日剩余天数；保养触发判断的唯一口径 |
| 保养状态 | `completionStatus` | `0` 表示未完成 |
| APS 安排日期 | `scheduleDate` | APS 本次实际安排保养的自然日，统一回填为当天00:00:00 |

APS MUST 只读取 `actualDate` 和 `completionStatus` 判断计划是否实际完成，MUST NOT 修改这两个字段。
排程成功事件只回填 `scheduleDate`；`scheduleDate` 仅记录 APS 最近一次安排的自然日，MUST NOT 作为运行态
加载或去重的过滤条件。已安排但设备侧尚未执行（`completionStatus=0` 且 `actualDate IS NULL`）的计划
SHALL 在后续滚动排程中重新加载，允许基于最新数据重新评估保养日期并覆盖回填 `scheduleDate`。

运行态精准计划查询 MUST 以排程引擎窗口起点T（`context.scheduleDate`）为边界，并同时满足：
工厂和年度匹配、未删除、`planDate >= T日00:00:00`、`completionStatus=0` 且 `actualDate IS NULL`；
MUST NOT 再包含 `scheduleDate IS NULL` 条件。年度计划完整性审计 MUST 继续基于
全年原始计划单独统计，MUST NOT 因运行态过滤把早于T日的计划误报为年度计划缺失。

用于日志和结果展示的有效到期日 MUST 按以下顺序解析：

```text
DUE_DATE → PLAN_DATE → 排程日 + DAYS_TO_DUE
```

精准计划预警和长期在机提前检查 MUST 直接读取 `daysToDue` 判断，MUST NOT 根据 `dueDate`、`planDate`
与排程T日重新计算剩余天数。`daysToDue` 为空时视为缺少触发依据，不进入保养触发判断；日期字段仅用于
计划日期过滤、日志和结果展示。

#### Scenario: 距到期天数直接取数据源字段

- **Given** 精度计划的 `daysToDue=20`，但 `dueDate` 或 `planDate` 与排程T日相差超过30天
- **When** APS 判断是否进入正常保养预警范围
- **Then** 系统 MUST 按20天判断为进入预警范围
- **And** 系统 MUST NOT 使用日期字段重新计算并覆盖 `daysToDue`

#### Scenario: 已完成计划不进入运行态

- **Given** 当年计划的 `actualDate` 非空或 `completionStatus` 不为 `0`
- **When** APS 加载保养计划
- **Then** 该计划 MUST NOT 进入 `maintenancePlanMap`

#### Scenario: 已安排但未执行的计划仍重新加载

- **Given** 当年计划已被上一轮排程回填 `scheduleDate`，但 `completionStatus=0` 且 `actualDate` 为空，且 `planDate >= T日`
- **When** APS 在后续滚动排程加载保养计划
- **Then** 该计划 MUST 重新进入 `maintenancePlanMap`
- **And** 系统 MUST NOT 因 `scheduleDate` 非空而排除该计划

### Requirement: 每台启用硫化机每个自然年度应有一条计划

系统 SHALL 检查每台启用硫化机在排程年度的精准计划条数。计划缺失或重复时，系统 MUST 同时写应用日志和
`T_LH_SCHEDULE_PROCESS_LOG`，但 MUST NOT 自动生成计划，也 MUST NOT 阻断其他机台排程。

#### Scenario: 年度计划缺失仅告警

- **Given** 启用机台 K1716 在排程年度没有精准计划
- **When** 基础数据加载完成
- **Then** 系统 MUST 写入“年度精准计划完整性检查”日志
- **And** 处理结果 MUST 为“年度精准计划缺失，仅告警不阻断”

#### Scenario: 年度计划重复仅告警

- **Given** 启用机台 K1501L 在排程年度存在两条有效记录
- **When** 基础数据加载完成
- **Then** 系统 MUST 写入重复告警
- **And** 系统 MUST NOT 自动删除或修改计划

## 参数

### Requirement: 保养参数必须兼容空值和非法历史数据

| 参数编码 | 业务含义 | 默认值 | 规则 |
| --- | --- | --- | --- |
| `SYS0307001` | 保养时长 | 7 小时 | 保养结束时间 = 开始时间 + 时长 |
| `SYS0307002` | 固定开始小时 | 8 | 最终候选日固定 08:00 开始 |
| `SYS0307003` | 正常保养预警天数 | 30 天 | 距到期天数小于等于该值进入正常保养判断 |
| `SYS0307005` | 是否允许周日保养 | 现有默认值 | `0` 顺延，`1` 允许 |
| `SYS0307006` | 节假日前禁排天数 | 2 天 | 节假日当天及此前 N 天禁排 |
| `SYS0307007` | 长期在机提前检查天数 | 3 天 | 进入该窗口后执行自然收尾预测 |
| `SYS0307009` | 胶囊预热时长 | 2.5 小时 | 支持小数小时 |

参数为空、格式非法或为负数时 MUST 使用业务默认值；每日最大保养台数按本规则固定为一台物理机，
不受历史 `SYS0307004` 大于一的配置影响。固定开始小时超出 0～23 时 MUST 回到默认 8 点。

## 正常保养

### Requirement: 正常保养必须在30天预警范围内由首个SKU真实收尾触发

系统 MUST 在续作、换活字块和新增规格切换入口统一调用中心保养服务。正常保养只有在精准计划未完成且
`daysToDue` 小于等于预警天数时才可安排。

保养候选日按物理机台首个 SKU 的最晚收尾时间确定：

```text
收尾时间 <= 当天08:00：当天为候选日，保养仍固定08:00开始
收尾时间 > 当天08:00：当天禁止保养，从下一自然日开始寻找
```

APS MUST NOT 因前 SKU 提前结束而把保养提前到 08:00 之前。

#### Scenario: SKU在08:00前收尾

- **Given** K1001 首个 SKU 在 07:59 收尾且计划进入30天预警
- **When** 系统安排保养
- **Then** 保养 MUST 在当天 08:00 开始

#### Scenario: SKU恰好08:00收尾

- **Given** K1001 首个 SKU 恰好在 08:00 收尾
- **When** 系统安排保养
- **Then** 当天 08:00 MAY 作为候选开始时间

#### Scenario: SKU在08:00后收尾

- **Given** K1001 首个 SKU 在 08:01 收尾
- **When** 系统安排保养
- **Then** 当天 MUST NOT 安排保养
- **And** 系统 MUST 从下一自然日 08:00 开始寻找可用日期

### Requirement: 保养候选日期必须循环校验禁排约束

每个候选日 MUST 依次校验：

1. 当天是否已有一台其他物理硫化机保养；
2. 是否为每月最后一天盘点日；
3. 是否为节假日当天或节假日前配置天数；
4. 周日参数是否允许安排。

任一约束命中时 MUST 向后顺延一个自然日并重新执行全部校验，日期运算 MUST 支持跨月、跨年。

#### Scenario: 每日已有其他机台保养

- **Given** 候选日已有另一台物理机占用保养额度
- **When** 当前机台选择保养日期
- **Then** 当前机台 MUST 顺延到下一可用日

#### Scenario: 月末盘点日

- **Given** 候选日为4月30日
- **When** 系统选择保养日期
- **Then** 4月30日 MUST 被跳过

#### Scenario: 节假日前两天

- **Given** 5月5日为节假日且禁排天数配置为2
- **When** 候选日为5月3日、5月4日或5月5日
- **Then** 候选日 MUST 顺延

#### Scenario: 周日参数控制

- **Given** 候选日为周日
- **When** `SYS0307005=0`
- **Then** 系统 MUST 顺延
- **When** `SYS0307005=1`
- **Then** 系统 MAY 在周日安排

#### Scenario: 跨年顺延

- **Given** 候选日为12月31日盘点日
- **When** 下一日满足其他约束
- **Then** 系统 MUST 正确顺延到下一年度1月1日08:00

## 长期在机强制下机

### Requirement: 长期在机必须先预测自然收尾再决定强制下机

连续在机超过30天且进入到期前检查天数时，系统 MUST 在续作目标量和起排时间明确后执行无副作用预测。
初始化阶段 MUST NOT 在没有预测结果时直接挂载强制保养窗口。

预测 MUST 复用逐班次真实产能，包括：

1. 当前最终目标量和模台数；
2. 机台班产、硫化时间和奇数班产修正；
3. 班次开停产管控；
4. 设备停机、清洗和已有保养不可生产区间；
5. 实际起排时间和班次时间边界。

预测过程 MUST NOT 修改月计划、日计划、胎胚库存、机台分配、结果列表或任何扣减账本。

若预测自然收尾时间不晚于最终候选保养日 08:00，系统 SHALL 等待自然收尾；若预测晚于候选时间，
或缺少可证明完成的数据，系统 MUST 强制下机。强制保养日期仍 MUST 执行每日一台、盘点日、节假日前限制和周日校验。

#### Scenario: 可自然收尾时不强制

- **Given** K1001 连续在机50天且距到期2天
- **And** 预测在候选日07:30自然收尾
- **When** 系统执行长期在机检查
- **Then** 系统 MUST 等待自然收尾

#### Scenario: 无法自然收尾时强制下机

- **Given** K1001 连续在机50天且距到期2天
- **And** 预测自然收尾晚于候选日08:00或预测无法完成
- **When** 系统执行长期在机检查
- **Then** 系统 MUST 在最终候选日08:00强制下机保养

## 单控物理机台

### Requirement: 单控L/R必须同步保养并只占一个每日额度

`K1501L` 和 `K1501R` 等 L/R 运行态 MUST 视为一台物理机：

1. 正常保养必须等待两侧均收尾或空闲，候选时间取两侧最晚收尾时间；
2. 长期在机预测必须覆盖所有活跃侧，任一侧无法完成即强制下机；
3. 任一侧触发保养时，两侧 MUST 同步挂载相同开始、结束和预热完成时间；
4. 每日额度只登记一次物理机台编码；
5. 两侧窗口分别保留各自精准计划主键，完成事件后分别回填；
6. 试制换模清除保养窗口时 MUST 同步清除配对侧并释放物理机台每日额度。

#### Scenario: L/R同步挂窗和回填

- **Given** K1501L、K1501R 各有一条年度精准计划
- **When** 任一侧触发保养
- **Then** 两侧 MUST 同步挂载窗口
- **And** 当日保养占用数 MUST 为1
- **And** 完成事件 MUST 按两个计划主键分别回填 `SCHEDULE_DATE`

## 时间轴与产能

### Requirement: 保养和预热必须完整占用不可生产时间轴

保养时间计算 MUST 为：

```text
保养开始 = 最终候选日 08:00
保养结束 = 保养开始 + 保养时长
最早开产 = 保养结束 + 胶囊预热时长
```

默认结果为 `08:00～15:00` 保养、`15:00～17:30` 胶囊预热，完整不可生产区间为 `08:00～17:30`。
结果保养摘要 MUST 继续展示真实保养结束时间15:00，产能计算和后续 SKU 就绪时间 MUST 使用17:30。

机台就绪时间 MUST 按保养是否进入当前固定班次范围分别判断：

```text
保养与当前固定班次范围相交，且当前就绪时间 < 预热完成：顺延到预热完成时间
保养在当前固定班次范围之外，且当前就绪时间 < 保养开始：保持原就绪时间
当前就绪时间落入保养开始～预热完成：顺延到预热完成时间
当前就绪时间 >= 预热完成：保持原就绪时间
```

不得直接取全部保养窗口中的最晚恢复时间，否则数周后的未来保养会提前锁死当前机台。生产跨越保养窗口时，
仍由统一不可生产区间扣减该窗口内产能，MUST NOT 允许在保养及预热期间生产。对于已经进入当前固定班次
范围的保养，后续 SKU 的结果开始时间 MUST NOT 早于预热完成时间，不能仅扣减班次产能而保留较早开始时刻。

设备停机、清洗、保养及预热重叠时 MUST 合并成时间区间并集，只扣减一次重叠时间；保养日期不因其他停机重叠而额外顺延。

### Requirement: 非试制正规换模与精度计划重叠时必须并行取最大完成时间

非试制 SKU 的正规换模窗口与精度计划完整占用区间重叠时，两项任务 SHALL 并行执行。精度计划开始小时、
保养时长和胶囊预热时长 MUST 分别复用 `SYS0307002`、`SYS0307001`、`SYS0307009`，正规换模时长
MUST 复用 `SYS0302009`，不得使用维保重叠专用4小时切换口径，也不得在正常换模时长后额外叠加首检小时。

当前序 SKU 已结束且换模可在精度计划开始时刻合法执行时，换模与精度计划 MUST 从同一时刻开始；
最终恢复生产时间 MUST 按以下公式计算：

```text
换模结束 = 精度计划开始 + 正规换模时长
精度恢复 = 精度计划结束 + 胶囊预热时长
最终恢复 = max(换模结束, 精度恢复)
```

默认参数下，精度计划及预热为08:00～17:30，正规换模为08:00～16:00，最终 MUST 在17:30恢复生产。
试制 SKU 继续沿用“重叠时清除精度计划窗口并优先早班换模”的既有规则；换活字块不适用本 Requirement。

#### Scenario: 默认参数下并行取9.5小时

- **Given** 非试制 SKU 在精度计划开始前已经收尾
- **And** 精度计划08:00开始、7小时保养、2.5小时胶囊预热
- **And** 正规换模耗时8小时
- **When** 精度计划与正规换模重叠
- **Then** 两项任务 MUST 在08:00并行开始
- **And** 换模 MUST 在16:00结束
- **And** 机台 MUST 在17:30恢复生产

#### Scenario: 保养后17:30开产

- **Given** 默认保养参数和2.5小时预热
- **When** 保养在08:00开始
- **Then** 15:00前 MUST 不可生产
- **And** 15:00～17:30预热期间产能 MUST 为0
- **And** 17:30后 MAY 恢复生产

#### Scenario: 保养占用班次的结果开始时间对齐

- **Given** 中班标准时间为14:00～22:00，精度保养及胶囊预热占用至17:30
- **And** 中班在17:30后仍有按有效产能计算出的计划量
- **When** 系统生成最终排程结果
- **Then** 中班结果开始时间 MUST 对齐为17:30
- **And** 对齐开始时间 MUST NOT 改变已计算出的班次计划量、日计划量、月计划量或胎胚库存账本
- **And** 若同一SKU结果同时包含保养前生产和17:30后的恢复生产，保养摘要及对应的独立/组合原因 MUST 绑定该SKU结果，MUST NOT 错挂到更晚的下一SKU

#### Scenario: 未来保养不提前阻断正常生产

- **Given** 机台当前就绪时间为7月17日22:00
- **And** 精度保养安排在8月5日08:00～17:30
- **And** 当前固定班次范围截至7月17日
- **When** 系统解析当前机台就绪时间
- **Then** 就绪时间 MUST 保持7月17日22:00
- **And** 8月5日08:00～17:30仍 MUST 作为不可生产区间参与后续滚动产能计算

#### Scenario: 与设备停机重叠

- **Given** 保养预热占用08:00～17:30，设备停机占用14:00～18:00
- **When** 系统计算不可生产时间
- **Then** MUST 按08:00～18:00并集扣减
- **And** MUST NOT 重复扣减14:00～17:30

## 日志、结果原因与回填

### Requirement: 关键判断必须同时写应用日志和排程过程日志

过程日志 SHALL 使用“精准计划保养判断”“精准计划最终安排”“年度精准计划完整性检查”等标题，
并包含机台、物理机台、保养类型、计划到期时间、距到期天数、候选时间、命中规则和处理结果。

日志 MUST 覆盖：是否进入30天预警、首个SKU收尾时间、每日额度、周日、盘点日、节假日前限制、
日期顺延、长期在机检查、预测自然收尾、强制下机、最终开始/结束/最早开产和因保养导致的就绪时间调整。
正规换模与精度计划重叠时，应用日志和过程日志 MUST 同时包含批次、排程日期、机台、物料、换模开始/结束、
精度开始、精度及预热完成、最终恢复时间和“换模+精度计划”组合原因。

同一机台、原就绪时间和最早开产时间组合在候选预演中重复计算时，过程日志 MUST 去重；不同原就绪时间
形成的真实调整仍 MUST 分别记录。

### Requirement: 保养结束时间所在班次必须备注精度计划或组合原因

本次排程实际触发并挂载精度保养窗口，且保养结束时间已进入当前排程固定班次范围时，系统 MUST 在
保养结束时间所在的标准班次原因分析中追加原因。未与正规换模重叠时使用固定文本：

```text
精度计划
```

非试制正规换模与精度计划实际重叠时 MUST 改为固定组合原因：

```text
换模+精度计划
```

组合原因与独立“精度计划”互斥，同一班次 MUST NOT 同时保留两项。

精度计划原因独立于“喷砂清洗+精度”等组合原因；即使与清洗或维修重叠，也 MUST 保留本 Requirement
确定的“精度计划”或“换模+精度计划”原因。最终结果阶段 MUST 先清理同机台中间结果残留的保养摘要、
独立“精度计划”和“换模+精度计划”原因，再优先把当前班次范围内
的窗口绑定到首条保养后结果；保养后没有后续 SKU 时，SHALL 绑定到该机台最后一条结果。同一运行侧的
一次保养 MUST 只绑定一条结果，其他清洗、维修和组合原因 MUST 保持不变。

保养结束时间尚未进入当前固定班次范围时，系统 MUST NOT 把未来保养摘要回退绑定到当前最后一条结果；
此时继续保留运行态窗口、过程日志和 `SCHEDULE_DATE` 回填，待后续滚动排程覆盖保养日期时再生成摘要和原因。

#### Scenario: 15:00结束落入中班

- **Given** 标准中班覆盖14:00～22:00
- **And** 精度保养在15:00结束
- **When** 系统保存最终排程结果
- **Then** 中班原因分析 MUST 包含“精度计划”

#### Scenario: 正规换模与精度计划重叠写组合原因

- **Given** 正规换模08:00～16:00与精度计划完整占用08:00～17:30重叠
- **When** 系统保存最终排程结果
- **Then** 保养结束所在中班原因分析 MUST 等于“换模+精度计划”
- **And** 同一班次 MUST NOT 残留独立“精度计划”

#### Scenario: 未来保养等待滚动窗口覆盖

- **Given** 当前结果班次只覆盖到7月17日
- **And** 精度保养安排在8月5日
- **When** 系统保存当前批次结果
- **Then** 当前结果 MUST NOT 回填8月5日的保养摘要或“精度计划”原因
- **And** 精度保养过程日志和 `SCHEDULE_DATE` 回填 MUST 保留
- **And** 后续滚动排程覆盖8月5日时 MUST 在对应班次生成摘要和“精度计划”原因

### Requirement: 完成事件必须按计划主键回填安排日期

排程结果成功保存后，完成事件监听器 MUST 遍历实际挂载的保养窗口，按 `precisionPlanId` 去重，
使用独立事务回填 `SCHEDULE_DATE=beginOfDay(maintenanceStartTime)`。安排到T+1/T+2时 MUST 分别回填
对应自然日的00:00:00。回填失败 MUST 输出明确错误日志，
但 MUST NOT 回滚已经成功保存的排程结果。

不得新增表或 XML；`SCHEDULE_DATE` 为现有数据库字段，只补齐实体映射并复用 MyBatis-Plus `updateById`。

#### Scenario: 保养后无后续SKU仍回填

- **Given** 机台已挂载有效保养窗口但没有后续排程结果
- **When** 系统发布排程完成事件
- **Then** 监听器 MUST 仍按窗口计划主键回填实际安排自然日00:00:00

#### Scenario: 回填失败不影响主排程

- **Given** 排程结果已成功保存
- **And** 精度计划安排日期回填发生异常
- **When** 事件发布器处理监听器异常
- **Then** 已保存排程结果 MUST 保持成功
- **And** 系统 MUST 记录包含批次、工厂和监听器的错误日志
