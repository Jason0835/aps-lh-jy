# 修复方案设计

## 方案结论

先按只读证据链排查并真实复跑；只有确认代码仍存在根因缺陷时，才做最小范围修复。当前优先怀疑点是日计划量准入与续作后置收口边界混用，而不是单个物料的特殊数据问题。

## 影响范围

需要重点检查以下现有逻辑：

1. 硫化排程入口与上下文初始化；
2. 续作和新增 SKU 分类逻辑；
3. 续作排产策略、日标准产量修正、最终结果收口；
4. 新增排产策略、候选机台选择、换模和首检结果写入；
5. 窗口无日计划 SKU 准入与提前生产规则；
6. 结果表、未排表、换模计划表、过程日志的落库链路。

## 设计思路

1. 以 2026-06-14 真实批次为准，不用其他日期推断。
2. 先查询现有最新结果；如证据不足，再调用 `/lhScheduleResult/execute` 复跑。
3. 对每个物料建立同一套对账字段：排程批次、机台、班次计划量、开始结束时间、换模/首检、未排原因、过程日志关键节点。
4. 对照主规格：
   - 续作日标准量只修正班次计划量，不限制当天排产量；
   - 新增提前生产只允许提前一天；
   - 窗口 T～T+2 无计划且未来计划在 19 日的 SKU 不应提前到 14 日排产。
5. 若确认缺陷，优先在共享入口修复，避免只对指定物料或单入口加特殊分支。

## 数据处理说明

排查期间只读查询排程结果和日志，不手工修改业务数据。若需要复跑，仅通过既有排程接口生成新批次，并以新批次的 `CREATE_TIME`、`LH_RESULT_BATCH_NO` 和 `IS_DELETE` 过滤结果，避免混用旧软删数据。

## 验证思路

验证必须覆盖：

1. `3302002326` 在 `K1905` 的续作班次是否按硫化余量和可用窗口继续排产；
2. `3302002319`、`3302001069` 是否存在第一天早班换模/首检以及中班生产结果；
3. `3302002366` 是否没有有效排程结果，并进入未排或被准入逻辑明确阻断；
4. 如果修复代码，需要补充 focused tests，并复跑 2026-06-14 进行表级对账。
