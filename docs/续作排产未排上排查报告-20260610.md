## 三、根因分析

问题发生在 S4.4 续作排产阶段，链条如下：

### 3.1 S4.3 — 续作分类正确

`classifyContinuousAndNewSkus()` 通过 MES 在机数据正确识别全部五个机台为续作，SKU 归入 `continuousSkuList`。

### 3.2 S4.3 — 收尾判定命中

`DefaultEndingJudgmentStrategy.isEnding()` 判定三个物料为收尾（规则2或规则3命中），`skuTag` 标记为 `ENDING`。

收尾判定规则：
- **规则2**：待排量 ≤ 多机台窗口总产能
- **规则3**：待排量 < 日产能

### 3.3 S4.4 — 续作收尾结果成功创建

流程日志（`t_lh_schedule_process_log`）显示五个机台均通过 `buildScheduleResult` 成功创建了续作收尾结果：

| 机台 | 物料 | specEndTime |
|------|------|-------------|
| K1305 | 3302001058 | 2026-06-09 12:32:44 |
| K2016 | 3302002186 | 2026-06-11 22:00:00 |
| K1708 | 3302002050 | 2026-06-11 22:00:00 |
| K1811 | 3302002050 | 2026-06-11 22:00:00 |
| K1923 | 3302002050 | 2026-06-11 22:00:00 |

### 3.4 S4.4 — `applyContinuousBlockToDailyQuota` 将结果裁剪为 0（核心问题）

`scheduleReduceMould()` → `syncContinuousDailyPlanQuota()` → `applyContinuousBlockToDailyQuota()` 对**所有续作结果**执行日计划额度裁剪。

关键代码（`ContinuousProductionStrategy.java:4063-4070`）：

```java
// 续作链路与新增链路保持一致：
// 收尾结果必须严格截断且不记超排
if (endingResult || (sku != null && sku.isStrictTargetQty())) {
    trimShiftPlanQty(result, shift.getShiftIndex(), consumedQty);
    if (endingResult) {
        continue;
    }
}
```

逻辑说明：
1. 对每个班次的计划量，调用 `SkuDailyPlanQuotaUtil.consumeRollingQuota()` 从日计划额度账本消费
2. 如果 `consumedQty < planQty`（额度不够），**收尾结果**直接截断为 `consumedQty`
3. 如果所有日期的账本额度均为 0 → `consumedQty = 0` → 所有班次被截断为 0 → `daily_plan_qty = 0`
4. `finalizeZeroPlanContinuousResults()` 将 `daily_plan_qty <= 0` 的结果从排程结果中移除，写入未排原因 `续作结果裁剪为0`

### 3.5 规则冲突的本质

收尾的语义是"把机器上剩余的尾量清掉"，尾量 = `max(胎胚库存, 月计划余量)`。这是物理需求口径。

但 `applyContinuousBlockToDailyQuota` 要求收尾结果必须由 dayN 日计划额度"买单"。当 dayN 额度已被 T 日排程完成量扣减、继承量扣减、或其他同物料 SKU 共享消费置零后，收尾结果被卡住。

**收尾不应当被 dayN 额度卡住。** 收尾的出发点是物理尾量，与 dayN 计划额度是两套体系。

### 3.6 3302002050 多机台降模的特殊情况

3302002050 有三台在机机台。`scheduleReduceMould` 中的多机台收尾降模逻辑：

```
targetQty = resolveContinuationDailyDemand()  → 收尾目标量
totalPlanQty = 三台机台合计排产量
if totalPlanQty > targetQty → 保留最少机台满足目标 → K1708 保留，K1811/K1923 裁剪
```

K1811 和 K1923 被正常降模释放。K1708 保留了 32 条，最终 `refreshContinuousEndingFlagByResult` 将其 `is_end` 标记从 1 翻转为 0（排产量未达收尾需求）。

### 3.7 S4.5 新增排产也无法补救

三个物料的续作结果被移除后，注册为续作释放机台。但随后在 S4.5 新增排产选机逻辑中：

- **3302001058**（寸口 R17.5）：K1305 在新增选机过滤器中被标记为"寸口不匹配"而被过滤。因为 K1305 的机台配置寸口可能不包含 R17.5（但在 MES 中实际在机说明模具已兼容），所以**连新增路径也无法选回 K1305**。
- **3302002186**（寸口 R24.5）：类似问题，R24.5 寸口机台范围被限定，K2016 在新增选机中同样被寸口过滤。

## 四、解决方案

### 4.1 核心思路

**收尾结果在 `applyContinuousBlockToDailyQuota` 中应保留完整计划量，不按 dayN 额度截断。**

收尾的物理尾量（`max(胎胚库存, 月计划余量)`）已经在上游通过 `upsizeEndingTargetQty` 收敛为目标量，此目标量不应被 dayN 额度二次截断。dayN 额度不足时，收尾结果应允许"超排"并记入 `shiftFillOverQty`，而不是被裁剪为 0。

### 4.2 修改方案

**修改文件**：`ContinuousProductionStrategy.java`

**修改位置**：`applyContinuousBlockToDailyQuota` 方法（约 4037-4082 行）

**修改内容**：

将当前逻辑：

```java
// 续作链路与新增链路保持一致：
// 收尾结果必须严格截断且不记超排
if (endingResult || (sku != null && sku.isStrictTargetQty())) {
    trimShiftPlanQty(result, shift.getShiftIndex(), consumedQty);
    if (endingResult) {
        continue;
    }
}
```

改为：

```java
// 续作链路与新增链路保持一致：
// 试制等严格目标量场景回裁后仍保留超排账本
// 收尾结果不按 dayN 额度截断：收尾的物理尾量已在上游收敛，dayN 额度不足时允许超排
if (!endingResult && sku != null && sku.isStrictTargetQty()) {
    trimShiftPlanQty(result, shift.getShiftIndex(), consumedQty);
}
if (endingResult) {
    // 收尾结果：允许 overQty 记入 shiftFillOverQty，不截断
    if (overQty > 0) {
        quota.setShiftFillOverQty(quota.getShiftFillOverQty() + overQty);
        totalShiftFillOverQty += overQty;
    }
    continue;
}
```

**改动说明**：
1. 收尾结果（`endingResult=true`）不再进入 `trimShiftPlanQty` 截断分支
2. 收尾的超过额度部分（`overQty`）记入 `shiftFillOverQty`，保持账本可追溯
3. 试制等严格目标量场景（`isStrictTargetQty=true` 但非收尾）仍保持原有截断逻辑

### 4.3 影响范围

| 影响点 | 说明 |
|--------|------|
| `applyContinuousBlockToDailyQuota` | 核心修改点，仅改收尾场景的裁剪行为 |
| `finalizeZeroPlanContinuousResults` | 不受影响，收尾结果保留后不会进入此清理 |
| `refreshResultSummary` | 不受影响 |
| `scheduleReduceMould` 多机台降模 | 不受影响，降模在 `syncContinuousDailyPlanQuota` 之前执行 |
| S4.5 新增排产 | 正面影响：收尾结果保留后机台被占用，不会错误释放给新增排产 |
| 胎胚库存调整 | 不受影响 |

### 4.4 不修改的场景

以下场景保持原有逻辑不变：
- **新增排产收尾**：新增路径的收尾结果仍按 dayN 额度截断（因为新增需要换模，必须确保 dayN 计划被遵守）
- **试制严格目标量**：`isStrictTargetQty=true` 的非收尾场景保持截断
- **多机台降模**：降模逻辑完全不受影响，仍在 `syncContinuousDailyPlanQuota` 之前运行
