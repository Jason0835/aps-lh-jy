# 硫化排程时间重叠场景处理机制分析

生成日期：2026-05-31

## 1. 明确结论

项目当前已经形成了比较完整的“生产窗口 + 不可生产窗口 + 切换窗口”处理框架，核心能力集中在 `ShiftCapacityResolverUtil`、`ShiftProductionControlUtil`、`MachineCleaningOverlapUtil`、`LhCleaningScheduleService`、`LhMaintenanceScheduleService` 以及新增、续作、换活字块三条排产策略中。

总体结论如下：

1. 设备停机、清洗、精度保养在最终班次分配和完工时间计算中，已经按不可生产窗口接入，主链能在多数场景下正确扣减产能并顺延完工时间。
2. 换模、换活字块与清洗的重叠语义已经下沉到策略层：切换窗口与清洗重叠时，不直接顺延换模或换活字块，而是在班次产能扣减阶段排除该重叠清洗窗口，避免同一段时间既算切换又算清洗损失。
3. 精度保养与换模、换活字块重叠时，现有实现采用专门的切换时间口径：保养结束后执行 4 小时切换，再追加 1 小时首检；非重叠保养则按保养结束后胶囊预热时间恢复生产。
4. 计划量计算主链仍存在一致性风险：`ShiftCapacityResolverUtil.resolveShiftCapacityWithDowntime` 对清洗和保养分段扣量时，没有统一按“停机 + 清洗 + 保养”的不可生产并集扣减，清洗与保养重叠时可能重复扣减产能。
5. 目标量预估、局部搜索预估与最终班次分配的窗口参数不完全一致，部分预估链路没有传入精度保养窗口，可能导致候选机台产能高估、目标量回裁不充分或局部搜索排序偏乐观。
6. 清洗、保养、停机的结果摘要字段是“生产区间实际命中的运行态窗口摘要”，不是源计划时间直抄，也不是换模窗口命中的所有窗口明细。该语义合理，但需要在排查和报表侧明确，否则容易误判为空值或漏算。

推荐改造方向是：不改变现有业务语义，不新增特殊分支，优先把不可生产窗口的并集计算能力收口到 `ShiftCapacityResolverUtil`，再同步修正目标量预估、局部搜索预估和换活字块停机预判的参数一致性。

## 2. 改造目标

本次分析建议后续改造聚焦四个目标：

1. 统一时间窗口重叠计算口径。
   设备停机、清洗、精度保养应在所有容量计算链路中使用一致的不可生产窗口并集，避免重复扣减或漏扣。

2. 统一计划量预估与最终分配口径。
   新增排产、续作排产、换活字块排产、目标量回裁、局部搜索候选估算都应使用同一类窗口参数，不能出现预估不扣保养、最终分配才扣保养的差异。

3. 明确实际开始时间和实际结束时间字段语义。
   精度保养、清洗、设备停机使用运行态窗口；换模、换活字块使用结果上的真实切换开始时间，结束时间按业务时长推导；结果摘要字段只记录生产区间内命中的窗口摘要。

4. 在不扩大改造范围的前提下提升可维护性。
   保持现有策略结构，优先改工具方法和参数传递，避免大范围重构，不引入新表、不新增无业务依据的特殊规则。

## 4. 现有机制总览

### 4.1 时间窗口分层

项目当前把时间窗口分为三层：

1. 班次可排窗口。
   由班次配置、工作日历、开产、停产共同决定。典型入口是 `ShiftProductionControlUtil.resolveEffectiveControl`。

2. 机台不可生产窗口。
   由设备停机、清洗、精度保养组成。典型入口是 `ShiftCapacityResolverUtil` 的停机、清洗、保养窗口集合计算。

3. 切换操作窗口。
   由换模、换活字块、首检组成。典型入口是 `DefaultMouldChangeBalanceStrategy`、`NewSpecProductionStrategy`、`TypeBlockProductionStrategy` 和 `ContinuousProductionStrategy`。

班次分配时的理论顺序是：

```text
班次自然窗口
    -> 开停产 / 工作日历裁剪
    -> 扣除设备停机
    -> 扣除清洗
    -> 扣除精度保养
    -> 按班产或硫化时间折算计划量
    -> 根据实际分配量反推班次结束时间
```

### 4.2 重叠判定规则

当前核心重叠判定基本使用严格相交：

```text
startA < endB && startB < endA
```

也就是说，两个窗口端点相接不算重叠。例如清洗 08:00-11:00，生产从 11:00 开始，不扣清洗时间；停机 08:00-10:00，换模从 10:00 开始，不算换模命中停机。

这个口径在 `MachineCleaningOverlapUtil`、`ShiftCapacityResolverUtil`、`ResultDowntimeSummaryUtil` 等类中基本一致。

### 4.3 运行态窗口优先于源计划

清洗和保养都存在“源计划”和“运行态窗口”的区别。

清洗源计划来自 `LhMouldCleanPlan`，但真正参与排程的是 `LhCleaningScheduleService` 生成的 `MachineCleaningWindowDTO`。清洗可能因为设备停机顺延，也可能因为喷砂禁排日、干冰工作时间窗口被调整。

保养源计划来自 `LhPrecisionPlan`，真正参与排程的是 `MachineMaintenanceWindowDTO`。保养窗口会根据日保养上限、周日、盘点日、节假日前阻塞、长期在机强制下机等规则生成。

设备停机当前基本直接使用源计划开始和结束时间，但在结果摘要中仍按生产区间求交后回填。

## 5. 分场景计算逻辑

### 5.1 精度计划重叠

#### 5.1.1 数据来源和初始化

精度计划当前对应年度精度保养计划，实体为 `LhPrecisionPlan`。基础数据加载入口是 `LhBaseDataServiceImpl.loadMaintenancePlan`，按以下条件加载：

1. 工厂编码匹配。
2. 年份匹配排程年份。
3. `completionStatus = 0`，即未完成。
4. `isDelete = 0`。

加载后按 `machineCode` 放入 `LhScheduleContext.maintenancePlanMap`。

保养运行态窗口由 `LhMaintenanceScheduleService` 生成，主要有两类触发：

1. 首个规格收尾后保养。
   `tryAttachMaintenanceAfterFirstEnding` 在机台第一个规格收尾后判断是否需要挂保养。

2. 长期在机强制保养。
   `tryAttachLongOnlineMaintenance` 和 `DataInitHandler.attachLongOnlineMaintenanceWindows` 对长期在机超过阈值且临近到期的机台挂保养窗口。

#### 5.1.2 保养窗口时间规则

当前默认规则是：

1. 保养开始时间：可用保养日期当天 08:00。
2. 保养时长：默认 7 小时。
3. 保养结束时间：默认 15:00。
4. 保养日期会受每日上限、周日、盘点日、节假日前阻塞规则影响。
5. 保养窗口写入 `MachineMaintenanceWindowDTO.maintenanceStartTime/endTime`。

#### 5.1.3 保养与生产重叠的计划量计算

班次分配时，保养窗口作为不可生产窗口传入 `ShiftCapacityResolverUtil.resolveShiftCapacityWithDowntime`。

现有逻辑大致为：

```text
班次有效秒数
    -> 扣设备停机并集
    -> 扣干冰清洗有效占用
    -> 扣保养有效占用
    -> 按班产或硫化时间折算最大计划量
    -> 扣喷砂清洗损失量
    -> 按模台数收敛
```

完工时间计算时，`resolveShiftPlanEndTime` 和 `resolveCompletionTimeWithDowntimes` 会把停机、清洗、保养收集为不可生产区间，排序合并后推进生产结束时间。

#### 5.1.4 保养与换模 / 换活字块重叠的开产规则

精度保养和换模、换活字块重叠时，当前不是简单把换模推到普通就绪时间之后，而是使用独立的维保重叠切换规则：

1. 如果切换参考时间落在保养结束前，认为命中保养重叠规则。
2. 切换开始时间以保养结束时间为基准。
3. 切换耗时使用 `MAINTENANCE_OVERLAP_SWITCH_HOURS`，默认 4 小时。
4. 首检再追加 `FIRST_INSPECTION_HOURS`，默认 1 小时。
5. 因此默认保养 08:00-15:00 时，重叠切换后的开产时间为 20:00。

非重叠保养恢复生产时，使用 `resolveMaintenanceResumeProductionTime`：

```text
恢复生产时间 = 保养结束时间 + 胶囊预热时间
```

默认保养 15:00 结束、胶囊预热 2.5 小时，则非重叠保养后可开产时间为 17:30。

#### 5.1.5 保养实际时间字段

保养实际开始 / 结束时间口径如下：

1. 运行态保养窗口：`MachineMaintenanceWindowDTO.maintenanceStartTime/endTime`。
2. 结果摘要字段：`LhScheduleResult.maintenanceStartTime/endTime`。
3. 摘要字段由 `ResultDowntimeSummaryUtil.fillDowntimeSummary` 按结果生产区间与保养窗口求交后回填。

需要注意：如果保养只影响换模窗口，但没有与生产区间重叠，结果摘要字段可能不记录该保养窗口。这不是必然漏算，而是当前字段语义决定的。

### 5.2 清洗计划重叠

#### 5.2.1 数据来源和运行态窗口

清洗计划来自 `LhMouldCleanPlan`，由 `LhBaseDataServiceImpl.loadCleaningPlan` 加载。运行态清洗窗口由 `LhCleaningScheduleService.buildScheduledCleaningWindowMap` 构建。

运行态窗口字段包括：

1. `cleanStartTime`：实际清洗开始时间。
2. `cleanEndTime`：清洗占用结束时间。
3. `readyTime`：模具可用时间。

喷砂和干冰的时间口径不同：

1. 干冰清洗：清洗时长和 ready 时长一致。
2. 喷砂清洗：`cleanEndTime` 使用“喷砂 + 首检”的总占用时长，`readyTime` 使用喷砂本体结束时间，避免后续换模、换活字块重复计算首检。

#### 5.2.2 清洗窗口调整规则

清洗运行态窗口生成时会处理以下规则：

1. 近收尾机台跳过清洗。
2. 干冰清洗开始时间归一到允许工作时间窗口。
3. 喷砂清洗尽量寻找前置可用日期，并避开喷砂禁排日。
4. 清洗窗口与设备停机重叠时，清洗开始时间顺延到停机结束，并循环处理后续停机。
5. 干冰和喷砂分别受日上限、班次上限控制。
6. 调整后的清洗窗口如果不在排程窗口内，不计入本次运行态占用。

#### 5.2.3 清洗与生产重叠的计划量计算

清洗进入班次产能计算时分两类处理：

1. 干冰清洗。
   以时间占用方式扣减可用秒数，再折算班次最大计划量。

2. 喷砂清洗。
   使用喷砂损失量折算，按重叠时长占喷砂总时长的比例计算损失，再从班次最大量中扣除。

当前工具方法在计算干冰和喷砂时，已经会剔除与设备停机重叠的部分，避免同一段时间同时作为停机和清洗重复扣量。

#### 5.2.4 清洗与换模 / 换活字块重叠

现有主链对清洗与换模、换活字块重叠的处理原则是：

1. 不因为清洗重叠直接顺延换模或换活字块。
2. 在班次产能计算前，通过 `MachineCleaningOverlapUtil.excludeOverlapWindows` 排除与切换窗口重叠的清洗窗口。
3. 对首个生产班次原因分析补充“模具清洗+换模”或“模具清洗+换活字块”。

这套处理避免了重复扣减：切换窗口已经占用时间，和切换重叠的清洗不应再从生产班次产能中扣一次。

#### 5.2.5 清洗实际时间字段

清洗实际开始 / 结束时间口径如下：

1. 运行态清洗窗口：`MachineCleaningWindowDTO.cleanStartTime/cleanEndTime/readyTime`。
2. 结果摘要字段：`LhScheduleResult.cleaningStartTime/endTime`。
3. 模具交替计划：`T_LH_MOULD_CHANGE_PLAN.PLAN_DATE` 和 `CHANGE_TIME` 使用运行态 `cleanStartTime`。

同样需要注意：如果清洗只与换模窗口重叠，不与生产区间重叠，结果摘要字段可能不记录该清洗；但模具交替计划仍会追加清洗类记录。

### 5.3 设备停机计划重叠

#### 5.3.1 数据来源和加载规则

设备停机计划来自 `MdmDevicePlanShut`，加载入口为 `LhBaseDataServiceImpl.loadDevicePlanShut`。查询规则是按排程窗口求区间重叠：

```text
beginDate <= 排程结束时间
endDate >= 排程开始时间
```

这说明只要停机计划与排程窗口有交集，就会被加载进入上下文。

#### 5.3.2 停机与生产重叠的计划量计算

设备停机是最基础的机台级不可生产窗口。`ShiftCapacityResolverUtil` 会先对停机窗口做以下处理：

1. 裁剪到当前班次有效窗口内。
2. 按开始时间排序。
3. 合并相交或相邻停机区间。
4. 计算停机并集秒数。
5. 从班次有效秒数中扣除。

这种处理可以避免同一机台多条停机计划重叠时重复扣秒。

#### 5.3.3 停机与清洗重叠

清洗运行态生成时，如果清洗窗口与设备停机重叠，清洗会顺延到停机结束。产能计算时，清洗损失还会剔除与停机重叠部分。

因此停机与清洗有两道保护：

1. 前置生成运行态清洗窗口时顺延。
2. 产能扣减时剔除与停机重叠的清洗秒数。

#### 5.3.4 停机与换模 / 换活字块重叠

换模均衡策略 `DefaultMouldChangeBalanceStrategy` 会在分配换模开始时间时检查切换窗口是否命中设备停机。如果命中，则把候选换模开始时间顺延到相关停机窗口的最晚结束时间。

换活字块策略 `TypeBlockProductionStrategy` 也有类似的停机顺延逻辑，会在确定换活字块开始时间前避开设备停机。

#### 5.3.5 停机实际时间字段

设备停机实际开始 / 结束时间口径如下：

1. 源停机窗口：`MdmDevicePlanShut.beginDate/endDate`。
2. 机台运行态摘要：`MachineScheduleDTO.shutdownStartTime/endTime` 当前保存命中的停机摘要。
3. 结果摘要字段：`LhScheduleResult.shutdownStartTime/endTime`，按结果生产区间与停机窗口求交后回填。

如果停机只影响换模起点，但没有与生产区间相交，结果摘要字段可能不记录该停机窗口。

### 5.4 换模 / 换活字块重叠

#### 5.4.1 换模开始时间规则

普通换模开始时间由 `DefaultMouldChangeBalanceStrategy.allocateMouldChange` 分配，主要规则如下：

1. 候选开始时间不得与设备停机窗口重叠。
2. 命中禁止换模时段时顺延到允许时段。
3. 夜班不发起换模。
4. 受早班 / 中班换模配额限制。
5. 返回的时间写入 `LhScheduleResult.mouldChangeStartTime`。

普通换模时长由配置解析，当前主链认为普通换模总时长已经包含首检时间，因此普通换模后的生产开始时间通常等于换模分配后的就绪时间。

#### 5.4.2 换活字块开始时间规则

换活字块由 `TypeBlockProductionStrategy` 和续作内部换活字块链路处理，主要规则如下：

1. 先按机台就绪时间、停机窗口、禁止换模时段确定允许切换开始时间。
2. 如果与精度保养重叠，采用维保重叠专用规则。
3. 如果不与精度保养重叠，按普通换活字块总时长计算开产时间。
4. 同样使用 `LhScheduleResult.mouldChangeStartTime` 记录真实切换开始时间。

#### 5.4.3 与清洗重叠

换模、换活字块与清洗重叠时，现有主链不顺延切换，而是排除与切换窗口重叠的清洗窗口，再进入生产班次产能扣减。

这说明当前业务语义是：清洗可以和切换操作重叠执行，至少在产能损失计算上不重复占用生产时间。

#### 5.4.4 与保养重叠

换模、换活字块与保养重叠时，采用保养结束后 4 小时切换 + 1 小时首检的专用口径。

这个口径已经同步到新增排产、续作内部换活字块、换活字块排产等主链，但局部搜索和部分估算链路仍需要重点核对参数传递是否一致。

#### 5.4.5 与设备停机重叠

切换窗口与设备停机重叠时，当前采取顺延切换开始时间的策略，不允许在设备停机期间发起切换。

需要注意，换活字块策略里部分停机预判使用固定换活字块总时长判断停机重叠。当实际命中维保重叠专用切换时长时，存在过度顺延的风险，需要通过用例确认。

## 6. 计划量与班次分配算法

### 6.1 班次最大计划量计算

当前班次最大计划量计算主入口是 `ShiftCapacityResolverUtil.resolveShiftCapacityWithDowntime`。核心公式分两类：

1. 有班产主数据：

```text
班次最大量 = 班产量 * 有效可生产秒数 / 班次总秒数
```

然后向下取整，并按模台数收敛。

2. 无班产主数据：

```text
班次最大量 = 有效可生产秒数 / 单锅硫化秒数 * 模台数
```

同样向下取整。

### 6.2 有效可生产秒数

有效可生产秒数由以下部分共同决定：

```text
班次自然秒数
    - 开停产 / 工作日历裁剪掉的秒数
    - 设备停机并集秒数
    - 干冰清洗有效占用秒数
    - 精度保养有效占用秒数
```

喷砂清洗不是直接扣秒，而是折算为计划量损失：

```text
喷砂损失量 = 喷砂配置损失量 * 班次内喷砂重叠秒数 / 喷砂窗口总秒数
```

### 6.3 班次实际分配量

新增、续作、换活字块分配班次计划量时，大体步骤一致：

1. 根据当前班次和生产开始时间解析有效管控窗口。
2. 调用 `resolveShiftCapacityWithDowntime` 得到该班最大可排量。
3. 再根据开停产比例进行收敛。
4. 按剩余目标量取最小值。
5. 通过 `resolveShiftPlanEndTime` 反推本班实际结束时间。
6. 写入 `classNPlanQty`、`classNStartTime`、`classNEndTime`。
7. 更新 `specEndTime` 和停机 / 清洗 / 保养摘要字段。

### 6.4 完工时间计算

`resolveShiftPlanEndTime` 的关键是按分配量换算所需净生产秒数，然后调用 `resolveCompletionTimeWithDowntimes` 推进时间。

推进过程中会收集以下不可生产窗口：

1. 设备停机窗口。
2. 清洗窗口。
3. 精度保养窗口。

这些窗口会排序合并，生产时间推进时遇到不可生产窗口会跳到窗口结束时间。

这一路径的并集口径相对完整，是当前完工时间计算比班次最大量计算更稳的地方。

## 7. 开产时间与实际时间字段口径

### 7.1 开产时间计算规则

| 场景 | 开产时间规则 |
| --- | --- |
| 普通新增换模 | 机台就绪时间 -> 换模均衡分配 -> 首检均衡后开产，普通换模总时长通常已包含首检 |
| 换活字块 | 机台就绪时间 -> 停机 / 禁换时段调整 -> 换活字块总时长 -> 开产 |
| 保养后非重叠生产 | 保养结束时间 + 胶囊预热时间 |
| 保养与换模 / 换活字块重叠 | 保养结束时间 + 4 小时切换 + 1 小时首检 |
| 清洗与换模 / 换活字块重叠 | 不直接顺延切换，排除重叠清洗窗口后进入生产分配 |
| 设备停机与切换重叠 | 切换开始时间顺延到停机结束后 |
| 开产模式 | 切换和生产不得早于开产班次有效开始时间 |
| 停产模式 | 班次有效结束时间按停锅时间前置规则裁剪 |

### 7.2 实际开始 / 结束时间总表

| 类型 | 实际开始时间 | 实际结束时间 | 落库 / 使用位置 |
| --- | --- | --- | --- |
| 精度保养 | `MachineMaintenanceWindowDTO.maintenanceStartTime` | `MachineMaintenanceWindowDTO.maintenanceEndTime` | 结果摘要 `maintenance_start_time/end_time` |
| 清洗 | `MachineCleaningWindowDTO.cleanStartTime` | `MachineCleaningWindowDTO.cleanEndTime` | 结果摘要、清洗类交替计划 |
| 设备停机 | `MdmDevicePlanShut.beginDate` | `MdmDevicePlanShut.endDate` | 结果摘要 `shutdown_start_time/end_time` |
| 换模 | `LhScheduleResult.mouldChangeStartTime` | 按换模总时长推导 | 换模类交替计划 |
| 换活字块 | `LhScheduleResult.mouldChangeStartTime` | 按换活字块总时长推导 | 换活字块交替计划 |
| 班次生产 | `classNStartTime` | `classNEndTime` | 排程结果班次字段 |
| 规格生产 | 首个有量班次开始时间 | `specEndTime` | 排程结果规格结束时间 |

### 7.3 结果摘要字段边界

`ResultDowntimeSummaryUtil.fillDowntimeSummary` 使用以下口径：

```text
生产区间 = 首个有排产量班次开始时间 -> specEndTime
摘要字段 = 生产区间与停机 / 清洗 / 保养窗口求交后的最早开始和最晚结束
```

这意味着：

1. 摘要字段不记录只发生在换模前、且未与生产区间重叠的窗口。
2. 同一结果命中多段同类窗口时，摘要字段会压缩为最早开始到最晚结束，不保留中间间隔。
3. 如果要做精确审计，应优先查询运行态日志、过程日志或后续新增明细输出，而不是把当前 6 个摘要字段解释为多段明细。

## 8. 现有潜在问题

### 8.1 P1：清洗与保养重叠可能重复扣减产能

问题位置：

1. `ShiftCapacityResolverUtil.resolveShiftCapacityWithDowntime`
2. `resolveDryIceAdjustedAvailableSeconds`
3. `resolveMaintenanceAdjustedAvailableSeconds`
4. `resolveCleaningLossQty`

现有逻辑已经剔除了清洗与设备停机重叠部分，也会对设备停机并集做合并。但保养扣减只剔除停机重叠，没有剔除已扣除的清洗窗口；喷砂损失也只剔除停机重叠，没有剔除与保养重叠部分。

影响：

1. 同一班次内清洗与保养重叠时，最大计划量可能被低估。
2. 干冰清洗与保养重叠时，可能同时扣清洗占用秒数和保养占用秒数。
3. 喷砂清洗与保养重叠时，可能先扣保养秒数，再扣喷砂损失量。
4. 最终分配量偏小，可能导致欠产、扩机、换模决策偏保守。

建议：

1. 在 `ShiftCapacityResolverUtil` 中新增统一的已扣窗口集合计算，而不是分别按清洗、保养独立扣。
2. 或者直接以 `停机 + 干冰清洗 + 保养` 的并集计算有效秒数，再单独计算喷砂损失时剔除该并集窗口。
3. 保持现有喷砂损失业务语义不变，只修正重叠部分的重复扣减。

### 8.2 P1：目标量预估忽略保养窗口

问题位置：

1. `TargetScheduleQtyResolver.refineTargetQtyByMachineCapacity`
2. `TargetScheduleQtyResolver.calculateMachineAvailableCapacityByShiftInWindow`
3. 新增、续作、换活字块策略调用目标量回裁的入口

当前最终班次分配会传入 `maintenanceWindowList`，但目标量预估链路部分方法没有传入保养窗口。这会导致：

1. 目标量回裁阶段认为机台能力更大。
2. 最终分配阶段因为扣保养而排不完。
3. 结果表现为预估计划量与实际班次计划量不一致。

建议：

1. 为目标量预估方法补充 `maintenanceWindowList` 参数。
2. 所有调用点优先从 `MachineScheduleDTO.getMaintenanceWindowList()` 传入。
3. 保持无保养窗口时行为不变。

### 8.3 P1：局部搜索估算与主链窗口口径不一致

问题位置：

1. `LocalSearchMachineAllocatorStrategy`
2. 局部搜索候选产能估算和完工时间估算

当前局部搜索估算已处理部分清洗窗口，但没有完整传入保养窗口；喷砂与切换重叠的处理也需要和主链“不顺延切换、排除重叠清洗窗口”的语义再对齐。

影响：

1. 局部搜索候选完工时间可能偏早。
2. 候选排序可能偏向实际会被保养占用的机台。
3. 与最终排产结果之间存在解释成本。

建议：

1. 局部搜索容量估算统一传入停机、清洗、保养三类窗口。
2. 清洗过滤统一复用 `MachineCleaningOverlapUtil.excludeOverlapWindows`。
3. 对喷砂重叠场景补充回归用例，锁定“不因喷砂顺延切换”的主链语义。

### 8.4 P1：干冰损失量配置可能未完整接入

问题位置：

1. `LhScheduleConfigResolver`
2. `LhScheduleParamConstant.DRY_ICE_LOSS_QTY`
3. `ShiftCapacityResolverUtil.resolveCleaningLossQty`

项目文档和参数常量中存在干冰损失量配置，但配置解析器需要确认是否完整接入。如果 `scheduleConfig` 存在但没有解析该参数，则 `context.getParamIntValue` 可能取不到数据库配置值，退回默认值。

影响：

1. 规则引擎配置的干冰损失量不生效。
2. 清洗计划量扣减与业务配置不一致。

建议：

1. 在 `LhScheduleConfigResolver` 中接入干冰损失量参数。
2. 增加参数解析单元测试。
3. 用一个干冰清洗跨班样例验证配置值变化会影响班次最大量。

### 8.5 P2：换活字块停机预判可能使用固定时长导致过度顺延

问题位置：

1. `TypeBlockProductionStrategy.resolveDowntimeAdjustedSwitchStartTime`

当前换活字块停机预判使用普通换活字块总时长判断切换窗口是否命中停机。若实际场景命中精度保养重叠专用规则，真实切换时长可能变为 4 小时 + 首检 1 小时，与普通换活字块总时长不同。

影响：

1. 如果普通换活字块总时长更长，可能把本不冲突的停机判为冲突。
2. 换活字块开始时间可能被过度顺延。

建议：

1. 将实际 `switchDurationHours` 作为参数传入停机预判方法。
2. 在确定是否命中维保重叠规则后，再用真实切换窗口复核停机冲突。
3. 补充“保养重叠 + 后续短停机”的换活字块用例。

### 8.6 P2：同机多清洗窗口缺少显式互斥

问题位置：

1. `LhCleaningScheduleService.buildScheduledCleaningWindowMap`
2. `LhCleaningScheduleService.scheduleCleaningWindow`
3. `ResultValidationHandler.appendCleaningMouldChangePlans`

当前清洗窗口主要受日上限、班次上限、停机顺延约束，但没有显式看到同一机台多条清洗窗口之间按时间互斥排队。产能扣减阶段通过区间合并可以避免生产时间重复扣，但模具交替计划可能追加多条时间重叠的清洗记录。

影响：

1. 生产产能未必重复扣，但清洗执行计划可能出现同机同时间段多清洗。
2. 交替计划可读性和现场执行解释成本较高。

建议：

1. 先通过真实数据确认是否允许同机左右模同时清洗。
2. 如果业务不允许同机重叠清洗，应在运行态窗口生成阶段按机台做互斥顺延。
3. 如果业务允许左右模并行清洗，应在交替计划和文档中明确并行语义。

### 8.7 P2：摘要字段压缩多段窗口，不能支持精确审计

问题位置：

1. `ResultDowntimeSummaryUtil.fillDowntimeSummary`
2. `LhScheduleResult` 的停机、清洗、保养摘要字段

当前摘要字段记录同类窗口的最早开始和最晚结束。如果同一结果生产区间命中多段清洗或多段停机，中间空档会被压缩。

影响：

1. 主结果表适合快速看是否命中窗口，不适合还原所有重叠细节。
2. 排查“为什么某个班少排”时，需要结合过程日志或运行态窗口。

建议：

1. 保持主表摘要字段语义不变，避免影响已有报表。
2. 如需精确审计，优先增加过程日志明细或独立诊断输出，不直接改变主表字段含义。

### 8.8 P3：部分注释和依赖组织可维护性不足

问题位置：

1. `DefaultCapacityCalculateStrategy`
2. `ResultValidationHandler`

可维护性问题包括：

1. 部分注释仍保留旧的维保后开产解释，容易与现行胶囊预热、维保重叠切换规则混淆。
2. `DefaultCapacityCalculateStrategy` 手工创建 `LhMaintenanceScheduleService`，当前能运行，但与 Spring 管理风格不一致。
3. `ResultValidationHandler` 中存在 `SimpleDateFormat` 使用，不符合项目当前规范。

建议：

1. 注释更新应随主逻辑修复一起做，避免只改注释。
2. 服务注入方式调整要评估 bean 生命周期，避免扩大改造。
3. 时间格式化规范问题可作为独立小任务处理，不和重叠算法修复混在一起。

## 9. 改进方案技术细节

### 9.1 统一不可生产窗口并集计算

推荐在 `ShiftCapacityResolverUtil` 内部增加小型私有方法，复用现有区间收集和合并逻辑：

```text
collectMergedIntervals(windowStart, windowEnd, plannedStops, cleaningWindows, maintenanceWindows, cleaningFilter)
```

建议拆成两类并集：

1. 时间占用并集：
   设备停机 + 干冰清洗 + 精度保养。

2. 喷砂损失剔除基准：
   设备停机 + 干冰清洗 + 精度保养中已经占用的区间。

计算方式：

```text
时间型可生产秒数 = 班次有效秒数 - 时间占用并集秒数
喷砂有效重叠秒数 = 喷砂窗口与班次窗口重叠秒数 - 喷砂窗口与时间占用并集交集秒数
```

这样可以保持喷砂按损失量扣减的业务语义，同时避免喷砂与保养、干冰与保养重复扣减。

### 9.2 目标量预估补齐保养窗口

推荐修改 `TargetScheduleQtyResolver`：

1. `refineTargetQtyByMachineCapacity` 增加 `maintenanceWindowList` 获取和传递。
2. `calculateMachineAvailableCapacityByShiftInWindow` 增加保养窗口参数。
3. 所有调用 `resolveShiftCapacityWithDowntime` 的预估位置都传入同一台机的保养窗口。

影响范围主要是方法签名和调用点，不需要新增表或 XML。

### 9.3 局部搜索预估对齐主链

推荐修改 `LocalSearchMachineAllocatorStrategy`：

1. 候选估算时传入 `machine.getMaintenanceWindowList()`。
2. 完工时间估算传入保养窗口。
3. 清洗与切换重叠过滤使用 `MachineCleaningOverlapUtil.excludeOverlapWindows`。
4. 保留现有喷砂原因分析能力，但不要让局部搜索采用与主链不同的切换顺延口径。

### 9.4 换活字块停机预判使用真实切换时长

推荐修改 `TypeBlockProductionStrategy`：

1. 将 `resolveDowntimeAdjustedSwitchStartTime` 的固定时长替换为参数传入。
2. 在已判断维保重叠后，用实际切换时长进行停机窗口复核。
3. 保持夜间禁换和开产模式收口逻辑不变。

这属于局部方法签名调整，不需要重构策略结构。

### 9.5 清洗同机重叠先审计再改造

清洗同机互斥涉及现场执行语义，不能直接假设。

建议先增加诊断能力：

1. 对同机运行态清洗窗口按开始时间排序。
2. 发现窗口重叠时输出诊断日志，包含机台、清洗类型、开始时间、结束时间、源计划 ID。
3. 用真实批次确认是否存在同机多清洗重叠。

如果业务确认不允许重叠，再在 `LhCleaningScheduleService` 生成运行态窗口时顺延后一个窗口，并重新校验日上限、班次上限和排程窗口边界。

## 12. 风险评估和应对措施

### 12.1 计划量变化风险

修正重复扣减后，部分清洗与保养重叠场景的班次最大量会增加。这是预期变化，但会影响最终落机结果。

应对措施：

1. 用单元测试锁定只修正重叠部分，不改变非重叠场景。
2. 选取真实批次对比修复前后机台级计划量变化。
3. 对新增排产、续作、换活字块分别抽样验证。

### 12.2 候选机台排序变化风险

目标量预估和局部搜索补扣保养后，候选机台能力会更接近真实分配结果，可能导致候选排序变化。

应对措施：

1. 输出候选机台容量诊断日志，包含是否扣保养窗口。
2. 对有保养计划的机台重点核对候选排序。
3. 确认变更只影响保养窗口命中的机台。

### 12.3 清洗互斥规则业务风险

同机多清洗窗口是否允许并行，依赖现场业务。未经确认直接顺延可能改变计划。

应对措施：

1. 第一阶段只做诊断，不直接改变清洗互斥语义。
2. 拿真实清洗计划和执行数据确认。
3. 确认后再做最小范围改造。

### 12.4 结果摘要解释风险

摘要字段不是多段明细，修复算法后仍可能出现摘要字段为空但换模计划存在清洗记录的情况。

应对措施：

1. 明确摘要字段语义。
2. 排查时联查运行态窗口和交替计划。
3. 如需精确明细，新增诊断输出，不改变现有摘要字段含义。

## 13. 推荐验收口径

后续实施完成后，建议按以下口径验收：

1. 非重叠场景排程结果不变。
2. 停机 + 清洗 + 保养重叠场景按不可生产并集扣减，不重复扣。
3. 喷砂损失只对有效生产冲突部分扣减，不对已被停机、保养、干冰占用的时间重复扣。
4. 目标量预估、局部搜索估算和最终班次分配对保养窗口口径一致。
5. 保养与换模 / 换活字块重叠时，开产时间仍保持“保养结束 + 4 小时切换 + 1 小时首检”的现行业务口径。
6. 清洗与换模 / 换活字块重叠时，仍保持“不顺延切换、排除重叠清洗窗口”的现行业务口径。
7. 结果摘要字段只作为生产区间命中窗口摘要，不作为全部窗口明细使用。
