# 硫化排程接口性能优化方案

## 一、问题概述

| 指标 | 当前值 |
|------|--------|
| 接口路径 | `POST /lhScheduleResult/execute` |
| SKU 数量 | 200+ |
| 机台数量 | 180+ |
| 当前响应时间 | ~2 分钟 |
| 期望响应时间 | < 30 秒 |

---

## 二、执行流程概览

排程接口采用模板方法模式，执行链路如下：

```
Controller (分布式锁)
  └─ LhScheduleServiceImpl.executeSchedule
       └─ AbsLhScheduleTemplate.execute (6 步模板)
            ├─ S4.1 PreValidationHandler     — 前置校验与数据清理
            ├─ S4.2 DataInitHandler           — 基础数据并行加载 + 机台状态初始化
            ├─ S4.3 ScheduleAdjustHandler     — SKU 聚集与余量计算
            ├─ S4.4 ContinuousProductionHandler — 续作排产
            ├─ S4.5 NewSpecProductionHandler  — 新增排产（主要瓶颈）
            └─ S4.6 ResultValidationHandler   — 结果验证与持久化
```

## 三、性能瓶颈定位与分析

### 瓶颈 1：`getOccupiedMouldCodes` 每次 SKU 全量遍历（严重）

**位置：** `DefaultMachineMatchStrategy.java:567`

**问题：** 每次调用 `matchMachines` 时都重新遍历整个 `machineAssignmentMap` 构建已占用模具集合。随排程推进，`machineAssignmentMap` 持续增长。

```java
// 当前实现：每个 SKU 都全量遍历
private Set<String> getOccupiedMouldCodes(LhScheduleContext context) {
    Set<String> occupied = new HashSet<>();
    for (Map.Entry<String, List<LhScheduleResult>> entry : context.getMachineAssignmentMap().entrySet()) {
        for (LhScheduleResult result : entry.getValue()) {
            if (shouldIgnoreReleasedContinuousPlaceholder(context, result)) {
                continue;
            }
            if (result.getMouldCode() != null) {
                occupied.add(result.getMouldCode());
            }
        }
    }
    return occupied;
}
```

**复杂度：** O(SKUs × 累计排程结果数) ≈ 200 × (逐渐增长到 500+) = 100,000+ 次遍历

---

### 瓶颈 2：`refreshPendingNewSpecSkuTypeCounts` 每轮全量扫描（严重）

**位置：** `NewSpecProductionStrategy.java:1040`

**问题：** 主循环每处理一个 SKU 都重新遍历整个 `newSpecSkuList` 统计各类型计数。

```java
// 在主循环第 370 行被调用，每个 SKU 进入时执行一次
while (iterator.hasNext()) {
    refreshPendingNewSpecSkuTypeCounts(context);  // ← O(N) per iteration
    SkuScheduleDTO sku = iterator.next();
    ...
}
```

**复杂度：** O(N²)，200 个 SKU 时 = 40,000 次遍历

---

### 瓶颈 3：机台排序比较器中重复计算 `resolveLatestAssignedEndTime`（中等）

**位置：** `DefaultMachineMatchStrategy.java:745`

**问题：** 在 `resolveCandidateWindowProfile` → `resolveCandidateReferenceTime` → `resolveMachineOccupiedEndTime` 链路中，`resolveLatestAssignedEndTime` 对每台机台的已分配结果做 stream().filter().map().max() 操作。虽有 `profileCache` 保护单次排序内不重复计算，但：
- 每个 SKU 都创建新的 `profileCache`
- 排序阶段外的 `traceMachineCandidates` 又重新计算一遍所有 profile

```java
private Date resolveLatestAssignedEndTime(LhScheduleContext context, String machineCode) {
    List<LhScheduleResult> assignedResults = context.getMachineAssignmentMap().get(machineCode);
    return assignedResults.stream()
            .filter(result -> Objects.nonNull(result) && ...)
            .map(LhScheduleResult::getSpecEndTime)
            .max(Date::compareTo)
            .orElse(null);
}
```

**复杂度：** 每次 SKU 排序时 O(candidates × assignedResultsPerMachine)，再加 traceMachineCandidates 重算

---

### 瓶颈 4：`traceMachineCandidates` 重度日志开销（中等）

**位置：** `DefaultMachineMatchStrategy.java:1612`

**问题：** 当 `PriorityTraceLogHelper.isEnabled(context)` 为 true 时，每次 `matchMachines` 都为 TOP 10 台候选机台：
1. 重新计算 `CandidateWindowProfile`（与排序阶段重复）
2. 构建约 2KB+ 的 StringBuilder 字符串
3. 计算 16 维排序指标

每个 SKU 调用一次，200 个 SKU = 200 次大字符串构建 + 2000 次 profile 计算。

---

### 瓶颈 5：DataInitHandler 机台停机匹配无索引（低~中等）

**位置：** `DataInitHandler.java:221`

**问题：** 为每台机台初始化停机信息时，遍历整个 `devicePlanShutList` 列表。

```java
for (Map.Entry<String, LhMachineInfo> entry : context.getMachineInfoMap().entrySet()) {
    ...
    // 内层循环：对每台机台遍历全部停机计划
    for (MdmDevicePlanShut planShut : context.getDevicePlanShutList()) {
        if (machineCode.equals(planShut.getMachineCode())) {
            ...
        }
    }
}
```

**复杂度：** O(machines × shutdownRecords) = 180 × N（N 为停机记录数）

---

### 瓶颈 6：`selectCandidateMachine` 内部多次 List 过滤（低~中等）

**位置：** `NewSpecProductionStrategy.java:1320`

**问题：** 在内层 while 循环中，每次尝试选下一台机台时都执行 `filterAvailableCandidatesByMachineType` 遍历候选列表两次（单控 + 普通），然后 `resolveCanFinishRemainingQtyFirst` 和 `resolveTailConcentratedSplitMachine` 再各自遍历一次并调用 `calcMachineAvailableCapacityInWindow`。

**复杂度：** 每次选机 O(candidates × 4) + 产能计算开销

---

### 瓶颈 7：数据初始化线程池偏小

**位置：** `LhDataInitExecutorProperties.java`

**问题：** 默认 corePoolSize=4、maxPoolSize=8，队列容量=32。而并行任务有 20+ 个（含依赖链），实际并发受限。当数据库连接充裕时，线程池成为初始化阶段的瓶颈。

---

## 四、优化方案

### 优化 1：增量维护已占用模具集合（预估收益：15~25%）

**思路：** 在 `LhScheduleContext` 中维护一个 `Set<String> occupiedMouldCodeSet`，每次排程结果写入 `machineAssignmentMap` 时同步更新该集合，`matchMachines` 直接读取。

**改造位置：**
- `LhScheduleContext` — 新增 `occupiedMouldCodeSet` 字段
- 所有向 `machineAssignmentMap` 添加结果的位置 — 同步 add mouldCode
- `DefaultMachineMatchStrategy.getOccupiedMouldCodes` — 改为直接返回 context 中的集合

**示例代码：**

```java
// LhScheduleContext 新增字段
private Set<String> occupiedMouldCodeSet = new HashSet<>(64);

// 新增辅助方法
public void addAssignedResult(String machineCode, LhScheduleResult result) {
    machineAssignmentMap.computeIfAbsent(machineCode, k -> new ArrayList<>()).add(result);
    if (result.getMouldCode() != null) {
        occupiedMouldCodeSet.add(result.getMouldCode());
    }
}
```

```java
// DefaultMachineMatchStrategy 改为：
private Set<String> getOccupiedMouldCodes(LhScheduleContext context) {
    return context.getOccupiedMouldCodeSet();
}
```

**风险：** 低。需确保所有写入 machineAssignmentMap 的入口都走统一方法。

---

### 优化 2：SKU 类型计数增量维护（预估收益：5~10%）

**思路：** 将 `refreshPendingNewSpecSkuTypeCounts` 从全量扫描改为增量更新。每次从列表移除 SKU 时按类型递减计数器。

**改造位置：**
- `NewSpecProductionStrategy.removeCurrentNewSpecSku` — 改为按类型递减
- 去掉主循环开头的 `refreshPendingNewSpecSkuTypeCounts(context)` 调用

**示例代码：**

```java
// 在 S4.5 开始前做一次全量统计（初始化）
refreshPendingNewSpecSkuTypeCounts(context);

// 主循环中删除该调用
while (iterator.hasNext()) {
    // refreshPendingNewSpecSkuTypeCounts(context);  ← 删除
    SkuScheduleDTO sku = iterator.next();
    ...
}

// removeCurrentNewSpecSku 改为增量递减
private void removeCurrentNewSpecSku(LhScheduleContext context,
                                     Iterator<SkuScheduleDTO> iterator,
                                     SkuScheduleDTO sku) {
    iterator.remove();
    context.getNewSpecTypeRuleBlockedMap().remove(sku);
    // 增量递减
    decrementSkuTypeCount(context, sku);
}

private void decrementSkuTypeCount(LhScheduleContext context, SkuScheduleDTO sku) {
    if (isTrialConstructionStage(sku)) {
        context.setPendingTrialNewSpecSkuCount(Math.max(0, context.getPendingTrialNewSpecSkuCount() - 1));
    } else if (isMassTrialSku(sku)) {
        context.setPendingMassTrialNewSpecSkuCount(Math.max(0, context.getPendingMassTrialNewSpecSkuCount() - 1));
    } else if (isSmallBatchSku(sku)) {
        if (hasPendingWindowQuotaSmallBatchDemand(sku)) {
            context.setPendingSmallBatchNewSpecSkuCount(Math.max(0, context.getPendingSmallBatchNewSpecSkuCount() - 1));
        }
    } else {
        context.setPendingFormalNewSpecSkuCount(Math.max(0, context.getPendingFormalNewSpecSkuCount() - 1));
    }
}
```

**风险：** 低。需确认计数器初始化时机正确。

---

### 优化 3：机台最新结束时间缓存（预估收益：10~15%）

**思路：** 在 `LhScheduleContext` 中维护 `Map<String, Date> machineLatestEndTimeCache`，每次新增排程结果时更新对应机台的最晚结束时间。避免 `resolveLatestAssignedEndTime` 的 stream 操作。

**改造位置：**
- `LhScheduleContext` — 新增 `machineLatestEndTimeCache` 字段
- 所有写入排程结果的位置 — 同步更新缓存
- `DefaultMachineMatchStrategy.resolveLatestAssignedEndTime` — 改为查缓存

**示例代码：**

```java
// LhScheduleContext 新增
private Map<String, Date> machineLatestEndTimeCache = new HashMap<>(256);

public void updateMachineLatestEndTime(String machineCode, LhScheduleResult result) {
    if (StringUtils.isEmpty(machineCode) || result == null
            || result.getDailyPlanQty() == null || result.getDailyPlanQty() <= 0
            || result.getSpecEndTime() == null) {
        return;
    }
    Date current = machineLatestEndTimeCache.get(machineCode);
    if (current == null || result.getSpecEndTime().after(current)) {
        machineLatestEndTimeCache.put(machineCode, result.getSpecEndTime());
    }
}
```

```java
// DefaultMachineMatchStrategy 改为：
private Date resolveLatestAssignedEndTime(LhScheduleContext context, String machineCode) {
    return context.getMachineLatestEndTimeCache().get(machineCode);
}
```

**风险：** 低。缓存逻辑简单，只做 max 比较。

---

### 优化 4：关闭或降级 PriorityTraceLog（预估收益：10~20%）

**思路：** 生产环境默认关闭 `priorityTraceLogEnabled`，仅在调试场景手动开启。或改为仅记录前 3 台（当前 TOP 10），减少计算和字符串构建开销。

**改造位置：**
- `LhScheduleConfig.priorityTraceLogEnabled` — 默认值改为 false
- 或 `traceMachineCandidates` 方法中 topN 从 `MACHINE_SORT_TRACE_TOP_N(10)` 降为 3

**额外优化：** `traceMachineCandidates` 中的 profile 计算与排序阶段的 `profileCache` 是独立的两份，可以将排序阶段的 profileCache 传入 trace 方法复用：

```java
// sortCandidates 后将 profileCache 传给 trace
private void sortCandidates(...) {
    Map<String, CandidateWindowProfile> profileCache = new HashMap<>(16);
    candidates.sort(buildMachineComparator(context, sku, matchResult, profileCache));
    // 排序后将 cache 传给 trace
    traceMachineCandidates(context, sku, matchResult, candidates, trace, profileCache);
}
```

**风险：** 无业务风险。仅影响日志输出。

### 优化 5：设备停机列表按机台预索引（预估收益：2~5%）

**思路：** 在 `DataInitHandler.buildStandardDataObjects` 之前，将 `devicePlanShutList` 按 machineCode 分组为 Map。

**改造位置：**
- `DataInitHandler.buildStandardDataObjects`

**示例代码：**

```java
private void buildStandardDataObjects(LhScheduleContext context) {
    // 停机计划按机台预分组
    Map<String, List<MdmDevicePlanShut>> shutdownByMachine = new HashMap<>(context.getMachineInfoMap().size());
    for (MdmDevicePlanShut planShut : context.getDevicePlanShutList()) {
        if (StringUtils.isNotEmpty(planShut.getMachineCode())) {
            shutdownByMachine.computeIfAbsent(planShut.getMachineCode(), k -> new ArrayList<>())
                    .add(planShut);
        }
    }

    for (Map.Entry<String, LhMachineInfo> entry : context.getMachineInfoMap().entrySet()) {
        String machineCode = entry.getKey();
        ...
        // 替换原内层循环
        List<MdmDevicePlanShut> machineShutdowns = shutdownByMachine.getOrDefault(machineCode, Collections.emptyList());
        for (MdmDevicePlanShut planShut : machineShutdowns) {
            if (dto.getPlanStopStartTime() == null
                    || (planShut.getBeginDate() != null && planShut.getBeginDate().before(dto.getPlanStopStartTime()))) {
                dto.setPlanStopStartTime(planShut.getBeginDate());
                dto.setPlanStopEndTime(planShut.getEndDate());
                dto.setStopType(planShut.getMachineStopType());
            }
            ...
        }
    }
}
```

**风险：** 无。纯等价替换。

### 优化 6：数据初始化线程池扩容（预估收益：3~8%）

**思路：** 将线程池核心/最大线程数提升至与并行任务数匹配，减少任务等待时间。

**改造位置：**
- 配置文件 `application.yml`（或 `LhDataInitExecutorProperties` 默认值）

**建议参数：**

```yaml
aps:
  lh:
    data-init:
      executor:
        core-pool-size: 12
        max-pool-size: 16
        queue-capacity: 8
        keep-alive-seconds: 30
```

**说明：** 核心线程 12 覆盖大部分独立任务，队列缩小至 8 迫使快速扩展到 maxPoolSize，避免任务堆积。需确认数据库连接池大小 ≥ maxPoolSize。

**风险：** 需确认数据库连接池容量足够，否则可能引起连接竞争。

### 优化 7：`selectCandidateMachine` 减少重复遍历（预估收益：3~5%）

**思路：** 在进入内层 while 循环前，一次性将候选列表拆分为单控和普通两个子列表并缓存，避免每次 `selectCandidateMachine` 都重新过滤。

**改造位置：**
- `NewSpecProductionStrategy` 主循环内，第 463 行 while 循环之前预分组

**示例代码：**

```java
// 进入 while 循环前预分组
List<MachineScheduleDTO> singleControlCandidates = new ArrayList<>(4);
List<MachineScheduleDTO> normalCandidates = new ArrayList<>(candidates.size());
for (MachineScheduleDTO candidate : candidates) {
    if (isSingleControlMachine(context, candidate.getMachineCode())) {
        singleControlCandidates.add(candidate);
    } else {
        normalCandidates.add(candidate);
    }
}

while (true) {
    // 直接使用预分组结果，按 excludedMachineCodes 过滤
    MachineScheduleDTO candidateMachine = selectCandidateMachineFromPreGrouped(
            context, sku, singleControlCandidates, normalCandidates,
            excludedMachineCodes, machineMatch, preferredTrialMachine, quantityPolicy);
    ...
}
```

**风险：** 低。需确认内层循环不会修改 candidates 列表本身。

### 优化 8：`resolveCanFinishRemainingQtyFirst` 提前短路（预估收益：2~3%）

**思路：** 该方法对每个候选机台调用 `calcMachineAvailableCapacityInWindow`（重计算），但结果只需要第一个满足条件的机台。由于候选已按优先级排序，可以限制搜索范围为前 N 台（如前 5 台）。

**改造位置：**
- `NewSpecProductionStrategy.resolveCanFinishRemainingQtyFirst`

**示例代码：**

```java
private MachineScheduleDTO resolveCanFinishRemainingQtyFirst(LhScheduleContext context,
                                                             SkuScheduleDTO sku,
                                                             List<MachineScheduleDTO> candidates,
                                                             Set<String> excludedMachineCodes) {
    ...
    int searchLimit = 5; // 仅搜索排序前 5 台
    int searched = 0;
    for (MachineScheduleDTO candidate : candidates) {
        if (candidate == null || ...) {
            continue;
        }
        searched++;
        if (searched > searchLimit) {
            break;
        }
        int machineCapacity = getTargetScheduleQtyResolver()
                .calcMachineAvailableCapacityInWindow(context, sku, candidate);
        if (machineCapacity >= remainingQty) {
            return candidate;
        }
    }
    return null;
}
```

**风险：** 极低概率下可能错过排序靠后但产能足够的机台，但该场景对排程质量影响可忽略。

## 五、优化优先级排序

| 优先级 | 优化项 | 预估收益 | 改动量 | 风险 |
|--------|--------|----------|--------|------|
| P0 | 优化 1：已占用模具增量维护 | 15~25% | 中 | 低 |
| P0 | 优化 4：关闭/降级 PriorityTraceLog | 10~20% | 小 | 无 |
| P1 | 优化 3：机台最新结束时间缓存 | 10~15% | 中 | 低 |
| P1 | 优化 2：SKU 类型计数增量维护 | 5~10% | 小 | 低 |
| P2 | 优化 5：停机列表按机台预索引 | 2~5% | 小 | 无 |
| P2 | 优化 6：线程池扩容 | 3~8% | 小 | 需确认连接池 |

| P3 | 优化 8：单机收完搜索限制 | 2~3% | 小 | 极低 |

**综合预估：** 全部优化落地后，排程时间可从 ~120 秒降至 40~60 秒（降幅 50~65%）。

## 六、实施建议

### 第一批次（低风险高收益，建议立即实施）

1. **优化 4** — 修改 `LhScheduleConfig.priorityTraceLogEnabled` 默认值为 false，或通过配置文件关闭。
2. **优化 2** — 将 `refreshPendingNewSpecSkuTypeCounts` 改为增量维护。
3. **优化 5** — 停机列表预分组。

### 第二批次（中等改动，核心收益）

4. **优化 1** — 已占用模具集合增量维护（需统一 machineAssignmentMap 写入入口）。
5. **优化 3** — 机台最新结束时间缓存。

### 第三批次（锦上添花）

6. **优化 6** — 线程池参数调整。
7. **优化 7** — 候选机台预分组。
8. **优化 8** — 单机收完搜索限制。

## 八、风险说明

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| 增量缓存与全量计算结果不一致 | 如果存在未通过统一入口修改 machineAssignmentMap 的路径 | 全局搜索所有写入点，统一收口 |
| 线程池扩容导致 DB 连接不足 | maxPoolSize 超过连接池上限 | 确认 HikariCP maxPoolSize ≥ 线程池 maxPoolSize |
| 关闭 TraceLog 影响排查能力 | 生产问题定位困难 | 保留配置开关，按需开启 |
| 搜索限制影响排程质量 | resolveCanFinishRemainingQtyFirst 限制后可能错过最优机台 | 限制值设为 5~8，覆盖绝大多数场景 |

---

## 九、补充建议（中长期）

1. **并行化 S4.5 新增排产**：将无依赖关系的 SKU 分组（如按结构/寸口段分桶），组内串行、组间并行。需解决机台竞争的线程安全问题，改动量大，建议作为下一阶段优化。

2. **SKU 排产预评估剪枝**：在进入主循环前，对每个 SKU 做轻量级机台可行性预判（仅检查寸口和模套），提前标记"完全无候选"的 SKU 跳过后续重计算。

3. **machineAssignmentMap 按结束时间有序存储**：改用 `TreeMap<Date, LhScheduleResult>` 或按 specEndTime 排序的 List，使 max 查询变为 O(1)。

4. **候选机台结果缓存**：对同结构/同寸口/同模套型号的 SKU 组，首次匹配后缓存候选列表骨架，后续 SKU 基于缓存做增量过滤（需排除模具占用变化）。
