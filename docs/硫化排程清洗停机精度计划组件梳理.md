# 硫化排程清洗停机精度计划组件梳理

生成日期：2026-06-03

本文档详细梳理硫化排程系统中涉及**清洗计划、设备停机计划、精度计划（精度保养）** 的相关方法、类、工具类等组件，包括数据模型、处理逻辑、时间窗口重叠处理机制。

## 一、清洗计划相关组件

### 1.1 数据模型

| 类 | 表 | 说明 |
|---|---|---|
| `LhMouldCleanPlan` | `T_LH_MOULD_CLEAN_PLAN` | 清洗计划源数据实体，字段包括 `lhCode`（机台编号）、`cleanType`（清洗类型）、`cleanTime`（清洗时间）、`leftRightMould`（左右模）、`dataSource` 等 |
| `MachineCleaningWindowDTO` | 无（运行态 DTO） | 清洗运行态窗口，由 `LhCleaningScheduleService` 生成 |

**`MachineCleaningWindowDTO` 关键字段语义：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `lhCode` | String | 机台编号 |
| `leftRightMould` | String | 左右模标识 |
| `cleanType` | String | 清洗类型（干冰/喷砂） |
| `mouldCode` | String | 模具编号 |
| `cleanStartTime` | Date | 运行态清洗开始时间（可能因停机顺延） |
| `cleanEndTime` | Date | 清洗占用结束时间（喷砂含首检时长，干冰等于 readyTime） |
| `readyTime` | Date | 模具可用时间（喷砂不含首检，干冰等于 cleanEndTime） |
| `dataSource` | String | 数据来源 |
| `remark` | String | 备注 |

### 1.2 核心类与方法

#### （1）LhBaseDataServiceImpl.loadCleaningPlan

- **所在类**：`com.zlt.aps.lh.service.impl.LhBaseDataServiceImpl`
- **作用**：从数据库加载清洗计划源数据
- **加载条件**：
  - 工厂编码匹配
  - 机台编号在已加载机台列表中（依赖 `machineInfoFuture` 先完成）
  - 清洗时间在排程窗口 `[startDate, endDate)` 内
  - 未删除（`isDelete = 0` 或 `null`）
- **上下文写入**：`context.setCleaningPlanList(...)`
- **排序**：按 `cleanTime` 升序

#### （2）LhCleaningScheduleService

- **所在包**：`com.zlt.aps.lh.service.impl`
- **注解**：`@Component`
- **核心方法**：`buildScheduledCleaningWindowMap(LhScheduleContext)`
- **作用**：将源计划 `LhMouldCleanPlan` 转换为运行态窗口 `MachineCleaningWindowDTO`，按机台编号分组返回 Map
- **处理规则**：
  1. 近收尾机台跳过清洗
  2. 干冰清洗开始时间归一到允许工作时间窗口（`DRY_ICE_WORK_START_TIME` ~ `DRY_ICE_WORK_END_TIME`）
  3. 喷砂清洗尽量寻找前置可用日期，避开喷砂禁排日
  4. **清洗窗口与设备停机重叠时**，调用 `resolveAdjustedCleaningStartTime` 将清洗开始时间顺延到停机结束（循环处理后续停机窗口）
  5. 干冰和喷砂分别受日上限（`DRY_ICE_DAILY_LIMIT`）、班次上限控制
  6. 调整后窗口不在排程窗口内的，不计入运行态占用
- **关键私有方法**：
  - `resolveAdjustedCleaningStartTime` — 循环检测清洗窗口是否与停机窗口重叠，命中则顺延到最晚停机结束时间
  - `scheduleCleaningWindow` — 单条清洗计划的运行态窗口生成

#### （3）DataInitHandler.attachCleaningPlanInfo

- **所在类**：`com.zlt.aps.lh.handler.DataInitHandler`
- **作用**：将运行态清洗窗口挂载到 `MachineScheduleDTO.cleaningWindowList`
- **处理逻辑**：
  1. 按机台编号匹配 + `LhSingleControlMachineUtil.isLeftRightCompatible` 左右模兼容过滤
  2. 按清洗开始时间升序排序
  3. 回填标志位：`hasDryIceCleaning`（干冰）、`hasSandBlastCleaning`（喷砂）
  4. 回填 `cleaningPlanTime`（取最早清洗开始时间）

#### （4）MachineCleaningOverlapUtil

- **所在包**：`com.zlt.aps.lh.util`
- **作用**：清洗窗口与换模/换活字块切换窗口的重叠判定和处理
- **核心方法**：

| 方法 | 说明 |
|---|---|
| `hasOverlap(cleaningWindowList, switchStartTime, switchEndTime)` | 判断清洗列表是否与切换窗口有任何重叠 |
| `hasBlockingOverlap(cleaningWindowList, switchStartTime, switchEndTime)` | 判断是否存在阻塞性重叠（喷砂使用端点相接也算重叠） |
| `isOverlap(cleaningWindow, switchStartTime, switchEndTime)` | 单条清洗窗口与切换窗口的**严格相交**判定：`startA < endB && startB < endA` |
| `isBlockingOverlap(cleaningWindow, switchStartTime, switchEndTime)` | 喷砂用 `cleanEndTime`（含首检），干冰用 `resolveEffectiveCleanEndTime`（优先 `readyTime`） |
| `excludeOverlapWindows(cleaningWindowList, switchStartTime, switchEndTime)` | **排除与切换窗口重叠的清洗窗口**，避免切换和清洗重复扣减产能 |
| `resolveDelayedSwitchStartBySandBlast(cleaningWindowList, switchStartTime, switchEndTime)` | 喷砂清洗导致切换开始时间顺延到喷砂结束 |
| `resolveEffectiveCleanEndTime(cleaningWindow)` | 喷砂取 `cleanEndTime`，干冰优先取 `readyTime` |

### 2.2 核心类与方法

#### （1）LhBaseDataServiceImpl.loadDevicePlanShut

- **作用**：加载与排程窗口有交集的停机计划
- **加载条件**：
  - `beginDate <= endDate(排程)` 且 `endDate >= startDate(排程)`，即区间重叠即加载
  - 工厂编码匹配
  - 未删除
- **时间范围**：使用 `calendarControlStartDate`（T-1日）到 `endDate`，覆盖喷砂前移和工作日历需求
- **上下文写入**：`context.setDevicePlanShutList(...)`
- **加载方式**：异步并行加载（`runDataInitTaskAsync`），无前置依赖

#### （2）DataInitHandler.buildStandardDataObjects 中的停机初始化

- **作用**：遍历停机列表，按机台匹配，取 `beginDate` 最早的停机记录写入 `MachineScheduleDTO`
- **写入字段**：
  - `planStopStartTime` — 最早停机开始时间
  - `planStopEndTime` — 对应停机结束时间
  - `stopType` — 停机类型编码
- **特殊处理**：
  - 临时性故障（`MachineStopTypeEnum.TEMPORARY_FAULT`）额外标记 `hasRepairPlan = true`
  - 记录最早维修时间 `repairPlanTime`（使用 `earlier` 方法取最早）

#### （3）MachineScheduleDTO 停机相关字段

| `stopType` | String | 停机类型编码 |
| `hasRepairPlan` | boolean | 是否有临时故障维修计划 |

## 四、共同使用的工具类和计算方法

### 4.1 ShiftCapacityResolverUtil — 班次产能解析核心工具

**所在包**：`com.zlt.aps.lh.util`

这是三类计划**共同依赖**的最核心计算工具，负责把"时间窗"换算成"可排量"或"完工时刻"。

#### 公开方法总览

| 方法 | 作用 | 涉及的计划类型 |
|---|---|---|
| `resolveShiftCapacityWithDowntime` | **班次最大计划量计算主入口**。依次扣停机→干冰清洗→保养→喷砂损失→模台数收敛 | 停机 + 清洗 + 保养 |
| `resolveShiftPlanEndTime` | **班次实际结束时间计算**。按分配量换算净生产秒数，推进时间跳过不可生产窗口 | 停机 + 清洗 + 保养 |
| `resolveCompletionTimeWithDowntimes` | **完工时间推导**。收集停机+清洗+保养的合并区间，生产时间遇到窗口跳到窗口结束 | 停机 + 清洗 + 保养 |
| `resolveDowntimeOverlapSeconds` | **不可生产并集秒数**。返回停机+清洗+保养的合并区间总秒数 | 停机 + 清洗 + 保养 |
| `resolveNetProductiveSeconds` | **净生产秒数** = 窗口总秒数 - 不可生产并集秒数 | 停机 + 清洗 + 保养 |
| `resolvePlannedStopOverlapSeconds` | 仅计算停机重叠秒数（裁剪+合并+统计） | 仅停机 |
| `resolveNetAvailableSeconds` | 仅扣停机后的净可用秒数 | 仅停机 |
| `resolveFirstSchedulableStartIgnoringCleaning` | 忽略清洗，只扣停机判断首个可排产班次 | 仅停机 |
| `resolveCompletionTimeWithPlannedStops` | 仅考虑停机的完工时间推导 | 仅停机 |
| `resolveShiftCapacity` | 基础班产折算（有班产按比例，无班产按硫化周期） | 通用 |
| `normalizeQtyToMouldMultiple` | 双模/多模残班向下收敛到模台数整数倍 | 通用 |
| `normalizeAllocatedShiftQty` | 统一收敛班次实际落点计划量 | 通用 |
| `resolveMachineMouldQty` | 解析机台模台数 | 通用 |
| `resolveRuntimeShiftCapacity` | 单控机台班产拆半 | 通用 |
| `resolveShiftDurationSeconds` | 解析班次总时长（秒） | 通用 |

#### 内部关键私有方法

| 方法 | 作用 |
|---|---|
| `collectMergedDowntimeIntervals` | 收集并合并"停机 + 清洗 + 保养"的所有不可生产区间 |
| `collectMergedPlannedStopIntervals` | 收集并合并停机区间（裁剪到时间窗 + 合并相交） |
| `collectMergedCleaningIntervals` | 收集并合并清洗区间（可按 `cleanType` 过滤干冰/喷砂） |
| `collectMergedMaintenanceIntervals` | 收集并合并保养区间 |
| `mergeIntervals` | 通用区间合并算法（按开始时间排序后合并交叉或相邻区间） |
| `resolveDryIceAdjustedAvailableSeconds` | 从可用秒数中扣除干冰清洗有效占用（剔除与停机重叠部分） |
| `resolveMaintenanceAdjustedAvailableSeconds` | 从可用秒数中扣除保养有效占用（剔除与停机重叠部分） |
| `resolveSandBlastLossQty` | 计算喷砂清洗损失量（按重叠时长折算班次产能） |
| `resolveEffectiveCleaningOverlapSeconds` | 单条清洗窗口的有效重叠秒数（剔除与停机重叠部分，避免重复扣减） |
| `resolveCleaningLossQty` | 汇总喷砂损失量 |
| `resolveIntervalDurationSeconds` | 统计区间列表总时长（秒） |
| `resolveIntervalIntersectionSeconds` | 两组区间的交集总秒数（用于识别"清洗已被停机覆盖"的重叠部分） |
| `intervalDurationSeconds` | 单个区间时长（秒） |

#### resolveShiftCapacityWithDowntime 计算流程

```text
1. resolveNetAvailableSeconds       → 班次有效秒数 - 停机并集秒数
2. collectMergedPlannedStopIntervals → 收集合并后的停机区间列表
3. resolveDryIceAdjustedAvailableSeconds → 再扣除干冰清洗有效秒数（剔除与停机重叠）
4. resolveMaintenanceAdjustedAvailableSeconds → 再扣除保养有效秒数（剔除与停机重叠）
5. resolveShiftCapacity             → 按折算公式得到班次最大量
6. resolveCleaningLossQty           → 计算喷砂损失量
7. Math.max(stopAdjustedQty - cleaningLossQty, 0) → 扣除喷砂损失
8. normalizeQtyToMouldMultiple      → 模台数整数倍收敛
```

#### collectMergedDowntimeIntervals 合并流程

```text
1. collectMergedPlannedStopIntervals   → 停机区间（裁剪+合并）
2. collectMergedCleaningIntervals      → 清洗区间（全部类型，裁剪+合并）
3. collectMergedMaintenanceIntervals   → 保养区间（裁剪+合并）
4. mergeIntervals(全部合并)            → 排序后合并所有交叉/相邻区间
```

## 五、时间重叠处理机制总览

### 5.1 窗口三层结构

```text
第一层：班次可排窗口
    → 班次配置 + 工作日历 + 开停产管控（ShiftProductionControlUtil）

第二层：机台不可生产窗口
    → 设备停机 + 清洗 + 精度保养（ShiftCapacityResolverUtil 统一处理）

第三层：切换操作窗口
    → 换模 + 换活字块 + 首检（各策略类处理）
```

### 5.2 重叠判定统一口径

所有核心重叠判定使用**严格相交**：

```text
startA < endB && startB < endA
```

端点相接不算重叠。例如清洗 08:00-11:00，生产从 11:00 开始，不扣清洗时间；停机 08:00-10:00，换模从 10:00 开始，不算换模命中停机。

该口径在 `MachineCleaningOverlapUtil`、`ShiftCapacityResolverUtil`、`ResultDowntimeSummaryUtil` 中保持一致。

### 5.3 运行态窗口优先于源计划

清洗和保养都存在"源计划"和"运行态窗口"的区别：

- **清洗**：源计划来自 `LhMouldCleanPlan`，真正参与排程的是 `LhCleaningScheduleService` 生成的 `MachineCleaningWindowDTO`。清洗可能因为设备停机顺延，也可能因为喷砂禁排日、干冰工作时间窗口被调整。
- **保养**：源计划来自 `LhPrecisionPlan`，真正参与排程的是 `MachineMaintenanceWindowDTO`。保养窗口会根据日保养上限、周日、盘点日、节假日前阻塞、长期在机强制下机等规则生成。
- **停机**：当前基本直接使用源计划 `MdmDevicePlanShut.beginDate/endDate`，但在结果摘要中仍按生产区间求交后回填。

### 5.4 两两重叠处理规则

| 重叠组合 | 处理策略 | 实现位置 |
|---|---|---|
| **停机 ↔ 清洗** | 双重保护：① 清洗窗口生成时顺延到停机结束（`resolveAdjustedCleaningStartTime` 循环检测）；② 产能扣减时剔除与停机重叠的清洗秒数（`resolveEffectiveCleaningOverlapSeconds`） | `LhCleaningScheduleService` + `ShiftCapacityResolverUtil` |
| **停机 ↔ 保养** | 保养扣减时剔除与停机重叠部分（`resolveMaintenanceAdjustedAvailableSeconds`） | `ShiftCapacityResolverUtil` |
| **停机 ↔ 换模/换活字块** | 切换开始时间顺延到停机结束后 | `DefaultMouldChangeBalanceStrategy` + `TypeBlockProductionStrategy` |
| **清洗 ↔ 换模/换活字块** | **不顺延切换**，排除与切换窗口重叠的清洗窗口（`excludeOverlapWindows`），避免切换和清洗重复扣减产能 | `MachineCleaningOverlapUtil` |
| **保养 ↔ 换模/换活字块** | 采用保养结束后 4 小时切换 + 1 小时首检的专用口径（`MAINTENANCE_OVERLAP_SWITCH_HOURS` + `FIRST_INSPECTION_HOURS`） | `LhMaintenanceScheduleService.shouldApplyMaintenanceOverlapSwitchRule` + `NewSpecProductionStrategy` |
| **保养 ↔ 非重叠生产** | 保养结束 + 胶囊预热时间（默认 2.5 小时）后恢复生产 | `resolveMaintenanceResumeProductionTime` |
| **清洗 ↔ 保养** | **当前存在 P1 风险**：干冰与保养可能重复扣秒，喷砂与保养可能重复扣量 | `ShiftCapacityResolverUtil` 中待修复 |

### 5.5 班次产能计算主流程

```text
班次自然秒数
    - 开停产/工作日历裁剪（ShiftProductionControlUtil）
    - 设备停机并集秒数（collectMergedPlannedStopIntervals → 合并 → 统计）
    - 干冰清洗有效占用秒数（剔除与停机重叠）
    - 精度保养有效占用秒数（剔除与停机重叠）
    = 有效可生产秒数

有班产主数据：班次最大量 = 班产量 × 有效可生产秒数 / 班次总秒数（向下取整）
无班产主数据：班次最大量 = 有效可生产秒数 / 单锅硫化秒数 × 模台数

    - 喷砂损失量（按有效重叠时长折算）
    - 模台数整数倍收敛
    = 最终班次最大计划量
```

### 5.6 完工时间计算流程

```text
1. 收集停机 + 清洗 + 保养所有窗口
2. 裁剪到时间窗
3. 排序合并为不重叠区间列表（mergeIntervals）
4. 从生产开始时间推进
5. 遇到不可生产区间 → 跳到区间结束时间
6. 净生产秒数消耗完毕时 → 返回完工时间
```

### 5.7 开产时间计算规则

| 场景 | 开产时间规则 |
|---|---|
| 普通新增换模 | 机台就绪时间 → 换模均衡分配 → 首检均衡后开产，普通换模总时长通常已包含首检 |
| 换活字块 | 机台就绪时间 → 停机/禁换时段调整 → 换活字块总时长 → 开产 |
| 保养后非重叠生产 | 保养结束时间 + 胶囊预热时间（默认 2.5 小时） |
| 保养与换模/换活字块重叠 | 保养结束时间 + 4 小时切换 + 1 小时首检（默认 20:00 开产） |
| 清洗与换模/换活字块重叠 | 不直接顺延切换，排除重叠清洗窗口后进入生产分配 |
| 设备停机与切换重叠 | 切换开始时间顺延到停机结束后 |
| 开产模式 | 切换和生产不得早于开产班次有效开始时间 |
| 停产模式 | 班次有效结束时间按停锅时间前置规则裁剪 |

### 5.8 实际开始/结束时间字段总表

| 类型 | 实际开始时间 | 实际结束时间 | 落库/使用位置 |
|---|---|---|---|
| 精度保养 | `MachineMaintenanceWindowDTO.maintenanceStartTime` | `MachineMaintenanceWindowDTO.maintenanceEndTime` | 结果摘要 `maintenance_start_time / end_time` |
| 清洗 | `MachineCleaningWindowDTO.cleanStartTime` | `MachineCleaningWindowDTO.cleanEndTime` | 结果摘要 `cleaning_start_time / end_time`、清洗类交替计划 |
| 设备停机 | `MdmDevicePlanShut.beginDate` | `MdmDevicePlanShut.endDate` | 结果摘要 `shutdown_start_time / end_time` |
| 换模 | `LhScheduleResult.mouldChangeStartTime` | 按换模总时长推导 | 换模类交替计划 |
| 换活字块 | `LhScheduleResult.mouldChangeStartTime` | 按换活字块总时长推导 | 换活字块交替计划 |
| 班次生产 | `classNStartTime` | `classNEndTime` | 排程结果班次字段 |
| 规格生产 | 首个有量班次开始时间 | `specEndTime` | 排程结果规格结束时间 |

## 六、已知潜在问题

| 优先级 | 问题 | 影响 | 涉及位置 |
|---|---|---|---|
| P1 | 清洗与保养重叠可能重复扣减产能 | 干冰+保养同时扣秒，喷砂+保养可能双重扣量；最终分配量偏小 | `ShiftCapacityResolverUtil.resolveShiftCapacityWithDowntime` |
| P1 | 目标量预估忽略保养窗口 | 预估能力偏大，最终分配排不完 | `TargetScheduleQtyResolver` |
| P1 | 局部搜索估算与主链窗口口径不一致 | 候选排序偏乐观 | `LocalSearchMachineAllocatorStrategy` |
| P1 | 干冰损失量配置可能未完整接入 | 规则引擎配置的干冰损失量不生效 | `LhScheduleConfigResolver` |
| P2 | 换活字块停机预判使用固定时长 | 命中保养重叠规则时可能过度顺延 | `TypeBlockProductionStrategy.resolveDowntimeAdjustedSwitchStartTime` |
| P2 | 同机多清洗窗口缺少显式互斥 | 交替计划可能出现同机同时段多清洗 | `LhCleaningScheduleService.buildScheduledCleaningWindowMap` |
| P2 | 摘要字段压缩多段窗口 | 不能支持精确审计 | `ResultDowntimeSummaryUtil.fillDowntimeSummary` |

## 七、数据流转全链路

```text
1. LhBaseDataServiceImpl.loadAllBaseData
    ├── loadDevicePlanShut          → context.devicePlanShutList
    ├── loadCleaningPlan            → context.cleaningPlanList
    └── loadMaintenancePlan         → context.maintenancePlanMap

2. DataInitHandler.buildStandardDataObjects
    ├── LhCleaningScheduleService.buildScheduledCleaningWindowMap
    │     └── 源计划 → 运行态窗口（停机顺延、日上限、班次上限）
    ├── attachCleaningPlanInfo
    │     └── 运行态窗口 → MachineScheduleDTO.cleaningWindowList
    ├── 停机初始化
    │     └── context.devicePlanShutList → MachineScheduleDTO.planStopStartTime/EndTime
    └── 胶囊使用次数等

3. DataInitHandler.attachLongOnlineMaintenanceWindows
    └── LhMaintenanceScheduleService.tryAttachLongOnlineMaintenance
          └── MachineScheduleDTO.maintenanceWindowList

4. 排产策略阶段（新增/续作/换活字块）
    ├── LhMaintenanceScheduleService.tryAttachMaintenanceAfterFirstEnding
    │     └── 首个规格收尾后动态挂载保养窗口
    ├── ShiftCapacityResolverUtil.resolveShiftCapacityWithDowntime
    │     └── 停机+清洗+保养 → 班次最大计划量
    ├── ShiftCapacityResolverUtil.resolveShiftPlanEndTime
    │     └── 分配量 → 班次实际结束时间
    ├── MachineCleaningOverlapUtil.excludeOverlapWindows
    │     └── 排除与切换重叠的清洗窗口
    └── LhMaintenanceScheduleService.shouldApplyMaintenanceOverlapSwitchRule
          └── 保养重叠 → 专用切换时间规则

5. 结果回填阶段
    └── ResultDowntimeSummaryUtil.fillDowntimeSummary
          └── 生产区间 × 停机/清洗/保养窗口 → 6 个摘要字段
```
