# 硫化停产管控与换模衔接修复方案

## 1. 明确结论

本次推荐方案是：在**不重构主流程**的前提下，围绕现有 `S4.4 续作/换活字块` 与 `S4.5 新增换模` 两条链路做最小改动修复，一次性解决以下 3 类问题：

1. `K1105 / 3302002654` 这类“新增情况，需要换模/换活字块”的场景，当前没有正确扣减换模时间，也没有纳入换模均衡。
2. 停产管控当前是按“停锅时间”直接截断，和业务要求的“**停锅时间 - 1 小时**”不一致。
3. 硫化开模时间、硫化停锅时间当前解析和展示仍以 `yyyy-MM-dd HH:mm:ss` 为主，需要统一为 `yyyy-MM-dd HH:mm`。

本次**不实现**以下内容：

- 第 4 条“达不到停产比例且有换模能力时，允许增模且不受在机 3 天限制”。

本次修复范围聚焦为：

- 停产截止时刻修正
- 停产收尾需求量链路核准
- 换活字块 / 新增换模分流修正
- 需要受换模约束的场景统一复用新增换模主链规则
- 开模/停锅时间格式统一

---

## 2. 改造目标

### 2.1 停产管控目标

严格按业务要求执行停产边界：

1. 硫化停锅时间由参数配置提供。
2. 根据硫化停锅时间推算停产班次。
3. 硫化计划量最晚只能排到**停锅时间前 1 小时**。
4. 停产收尾需求量按“**停产日月计划量（含损耗）与胎胚库存取大者**”计算，确保胎胚尽量收尾。

### 2.2 换模/换活字块目标

修复 `K1105 / 3302002654` 这类场景下：

1. 结果不能只在结果表里标记为换模/换活字块，而前置时间轴没有扣换模能力。
2. 需要受换模规则约束的场景，必须占用换模均衡能力。
3. 需要首检约束的场景，必须走已有首检分配链路。

### 2.3 时间格式目标

统一硫化开模时间、硫化停锅时间的输入/输出口径为：

```text
yyyy-MM-dd HH:mm
```

## 4. 现状问题分析

### 4.1 停产截止时刻实现错误

当前 `DefaultProductionShutdownStrategy.applyStopProductionControl(...)` 的逻辑是：

1. 若班次开始时间不早于停锅时间，则整班不可排
2. 若班次结束时间晚于停锅时间，则直接截到停锅时间

这意味着当前系统执行的是：

```text
排到 stopPotTime
```

而业务要求是：

```text
排到 stopPotTime - 1 小时
```

因此这不是参数问题，也不是结果层格式问题，而是**停产班次执行边界实现错误**。

### 4.2 停产收尾需求量与停产执行边界未闭环

`ScheduleAdjustHandler` 已经按下面逻辑计算停产收尾需求量：

```text
max(停产日计划量(含损耗), 胎胚库存)
```

这条逻辑方向是对的，但如果班次执行边界仍按 `stopPotTime` 截断，就会出现：

1. 目标量按停产收尾口径拉高
2. 班次可排时间却多放了 1 小时

最终排程结果会高于业务允许的真实停产窗口。

### 4.3 `K1105 / 3302002654` 场景本质是分流错误与规则绕路

当前问题不是单点字段没赋值，而是链路绕路：

1. SKU 在 `S4.4` 被提前命中换活字块路径
2. 没有进入 `S4.5` 新增换模主链
3. 没有调用 `allocateMouldChange(...)`
4. 没有占用换模均衡名额
5. 没有完整复用首检/停产/日历/维保/停机等统一规则

结果就是：

1. 结果表里像是发生了切换
2. 但换模时间扣减和换模均衡并没有真实生效

### 4.4 时间格式问题是“解析 + 展示”双口径问题

当前 `parseControlTime(...)` 强依赖 `yyyy-MM-dd HH:mm:ss`，而不是 `yyyy-MM-dd HH:mm`。  
如果只改日志格式，不改解析逻辑，会导致：

1. 参数仍然要求秒级
2. 外部看起来像支持分钟级
3. 输入输出口径不一致

因此必须同时处理：

1. 参数解析口径
2. 日志/文案/输出口径

## 5. 设计思路

## 5.1 总体策略

本次按照“**保留主流程、统一规则入口、收紧错误分流**”处理：

1. 不改 `S4.4 -> S4.5` 的大框架
2. 不重写新增换模主链
3. 不新增并行排程流程
4. 优先把错误场景导回已有正确链路

## 5.2 停产边界统一设计

新增统一概念：

```text
停产执行截止时刻 = curingStopPotTime - 1小时
```

规则设计：

1. `curingStopPotTime` 仍然保留原始参数语义
2. 停产班次识别仍基于原始停锅时间
3. 班次可排产窗口、最终有效截止时刻统一改为“停锅前 1 小时”

这样可以避免：

1. 改坏停产班次识别
2. 影响已有班次推算逻辑
3. 把“停锅时间”和“执行截止时间”混成一个字段

## 5.3 停产需求量设计

保留现有需求量公式：

```text
停产需求量 = max(含损耗停产日计划量, 胎胚库存)
```

本次不重写这条算法，只做两件事：

1. 核准它确实用于停产收尾目标量
2. 保证后续产能分配严格受“停锅前 1 小时”约束

## 5.4 换活字块 / 新增换模分流设计

### 原则

换活字块只处理真正符合换活字块业务语义的场景。  
凡是业务要求：

1. 扣减换模时间
2. 占用换模均衡名额
3. 统一受首检分配约束
4. 完整受停产管控约束

则必须复用新增换模主链能力，不能继续走当前普通换活字块的自算旁路。

### 最小改动做法

1. 先收紧 `TypeBlockProductionStrategy` 普通候选命中范围
2. 把本应走新增换模的场景留给 `NewSpecProductionStrategy`
3. 对确实仍需在 `S4.4` 衔接处理但又要求换模约束的少量场景，复用新增链路中的：
   - `allocateMouldChange(...)`
   - `allocateInspection(...)`
   - 首个可开产时刻计算
   - 目标量收敛

## 5.5 时间格式设计

时间格式改造原则：

1. 只改“硫化开模时间 / 硫化停锅时间”相关口径
2. 不全局替换所有日期时间格式方法
3. 兼容历史秒级配置时要审慎处理，避免运行中断

推荐口径：

1. 主格式：`yyyy-MM-dd HH:mm`
2. 如需兼容历史值，可同时兼容 `yyyy-MM-dd HH:mm:ss`
3. 对外文案统一写成 `yyyy-MM-dd HH:mm`

## 6. 详细实施步骤

## 6.1 第一步：补定向回归用例

先补测试，再改实现，固定本次业务口径。

### 用例 A：停锅前 1 小时截断

场景：

- 停锅时间：`2026-05-07 22:01`

断言：

1. 停产班次仍能正确识别
2. 实际可排截止时刻应为 `2026-05-07 21:01`
3. `21:01` 后不能再有新增可排产时间

### 用例 B：停产需求量取大者且考虑损耗

断言：

1. 停产日计划量会按损耗放大
2. 若胎胚库存更大，则取胎胚库存
3. 若含损耗停产日计划量更大，则取该值

### 用例 C：`K1105 / 3302002654` 分流修复

断言：

1. 不再误走当前普通换活字块旁路
2. 会进入受换模均衡控制的正确链路
3. `mouldChangeStartTime`、开产时间、换模计划时间轴一致

### 用例 D：开模/停锅时间分钟级格式

断言：

1. `yyyy-MM-dd HH:mm` 可正常解析
2. 开模/停锅时间相关日志或返回口径符合分钟级格式

## 6.2 第二步：引入停产执行截止时刻

改造 `DefaultProductionShutdownStrategy`：

1. 在停产管控内部统一计算：

```text
effectiveStopCutoffTime = curingStopPotTime - 1小时
```

2. `applyStopProductionControl(...)` 中所有对停锅时间的比较，改为对该截止时刻比较
3. 保留原始 `curingStopPotTime` 用于停产班次定位与日志语义

这样可以把“停锅参数”和“执行截止时间”解耦。

## 6.3 第三步：复核停产需求量入口

复核 `ScheduleAdjustHandler` 中：

1. `resolveStopDayPlanQty(...)`
2. `resolveLossIncludedStopDayPlanQty(...)`
3. `resolveStopProductionDemandQty(...)`

需要确认的点：

1. 是否确实取停锅所属自然日的 `DAY_n`
2. 损耗折算是否满足当前月计划数据结构语义
3. 停产模式下后续目标量是否全部使用该需求量

如没有逻辑错误，仅补注释和测试，不扩大改动。

## 6.4 第四步：收紧换活字块普通候选分流

改造 `TypeBlockProductionStrategy`：

1. 检查普通候选路径的命中规则
2. 减少“同规格/同胎胚描述”导致的误入
3. 把需要完整换模规则的场景留给 `S4.5`

重点是：

1. 不要只改结果构建位置
2. 要在候选命中阶段就把错误分流收住

## 6.5 第五步：统一复用新增换模主链规则

若业务上某些 `S4.4` 衔接场景仍需保留在当前阶段处理，则在 `TypeBlockProductionStrategy` 中最小复用：

1. `allocateMouldChange(...)`
2. `allocateInspection(...)`
3. `ShiftProductionControlUtil.resolveFirstSchedulableStartIgnoringCleaning(...)`
4. `TargetScheduleQtyResolver.refineTargetQtyByMachineCapacity(...)`

这样保证以下规则一致：

1. 换模能力扣减
2. 首检能力占用
3. 停产截止约束
4. 工作日历/维保/停机/清洗窗口处理

## 6.6 第六步：统一结果时间轴语义

保证以下字段统一使用真实切换开始时间：

1. `LhScheduleResult.mouldChangeStartTime`
2. `LhMouldChangePlan.planDate`
3. `LhMouldChangePlan.changeTime`

避免出现：

1. 结果时间一套
2. 换模计划时间一套

## 6.7 第七步：调整开模/停锅时间格式

### 解析层

调整 `parseControlTime(...)`：

1. 主格式支持 `yyyy-MM-dd HH:mm`
2. 如需兼容旧值，再兼容 `yyyy-MM-dd HH:mm:ss`
3. 错误提示统一为分钟级格式要求

### 展示层

统一以下输出口径：

1. 开模时间日志
2. 停锅时间日志
3. 停产班次推算日志
4. 相关错误文案

不建议全局替换 `formatDateTime(...)`。

## 7. 数据处理说明

## 7.1 停产需求量数据

来源：

1. 月计划停产日 `DAY_n`
2. 月计划需求 / 总量折算出的损耗系数
3. SKU 分摊后的胎胚库存

计算口径：

```text
demandQty = max(lossIncludedStopDayPlanQty, embryoStock)
```

## 7.2 停产执行窗口数据

来源：

1. `curingStopPotTime`
2. 班次窗口
3. 工作日历

执行口径：

```text
actualCutoffTime = curingStopPotTime - 1小时
```

## 7.3 换模能力数据

来源：

1. `dailyMouldChangeCountMap`
2. 早班 / 中班换模能力限制
3. 禁止换模时段

规则：

只要业务要求“真实扣减换模时间”，就必须通过 `allocateMouldChange(...)` 占用名额。

## 7.4 结果与计划回写

需要保证：

1. 结果表与换模计划表时间一致
2. 结果的 `scheduleType / isTypeBlock / isChangeMould` 语义一致
3. 停产模式下最终结果不会晚于“停锅前 1 小时”

## 8. 边界场景

必须重点验证以下场景：

1. 停锅时间落在班次中段，是否正确截到“停锅前 1 小时”
2. 停锅时间落在夜班，是否仍按当前确认口径处理

8. 开模/停锅时间是否支持 `yyyy-MM-dd HH:mm`
9. 历史秒级参数是否需要兼容读取

## 9. 风险点

## 9.1 结果变化风险

停产截止改成“停锅前 1 小时”后，末班可排量一定会下降，部分原先能排上的数量会被截掉，这是业务预期内变化。

## 9.2 分流调整风险

收紧 `TypeBlockProductionStrategy` 后，原本在 `S4.4` 生成的部分结果可能会转入 `S4.5`，对应：

1. 排程类型可能变化
2. 换模时间占用时点可能变化
3. 换模均衡计数可能变化

## 9.3 时间格式兼容风险

若直接改为只接受 `yyyy-MM-dd HH:mm`，历史仍传秒级值的调用方可能报错。  
因此实施时要先确认是否保留秒级兼容。

## 9.4 范围控制风险

本次已明确：

1. 不实现“达不到停产比例且有换模能力时，允许增模且不受在机 3 天限制”
2. 不做大范围重构
3. 不改无关排程规则

开发时必须严格控 scope。
