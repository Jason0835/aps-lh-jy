2. 排完119条 → `shouldContinueTypeBlockOnNextMachine()` → `return false`
3. 排产结束 → 剩余181条没有任何后续处理
4. `NewSpecProductionStrategy` → 3302002654 不会被再次排产（已被换活字块阶段处理过）

# 解决方案

## 问题一：换活字块SKU需要接入多机台分配

### 改造目标

换活字块排产完成后，如果SKU窗口目标量未排满，需要将剩余量转入新增排产逻辑，继续分配换模机台。

### 现有逻辑影响点

| 类 / 方法 | 位置 | 说明 |
|-----------|------|------|
| `TypeBlockProductionStrategy.shouldContinueTypeBlockOnNextMachine()` | ~L1076 | 当前 `isSingleMachine=true` 直接返回false |
| `TypeBlockProductionStrategy.scheduleTypeBlockChange()` | ~L108 | 主循环，排完后没有"转新增排产"机制 |
| `NewSpecProductionStrategy` | 新增排产入口 | 需要接收换活字块的剩余量 |
| `ScheduleAdjustHandler` 或调度编排层 | 排产阶段编排 | 换活字块 → 新增排产的衔接 |

### 设计思路

**方案**：换活字块排产完成后，如果 SKU 的窗口目标量未满，将剩余量注入到新增排产的待排SKU列表中，让 `NewSpecProductionStrategy` 接力分配换模机台。

具体步骤：
1. 在 `TypeBlockProductionStrategy` 排产完成后，检查 SKU 是否还有未排完的窗口目标量
2. 如果有剩余，构建一个新增排产的 SkuDTO（标记来源为换活字块接力），写入 `context.getNewSpecSkuList()`
3. `NewSpecProductionStrategy` 正常流程排产该SKU（走换模流程）
4. 排产时需排除已被换活字块占用的机台（K2024），避免重复

### 详细实施步骤

1. **`TypeBlockProductionStrategy` 修改**：
   - 在 `scheduleTypeBlockChange()` 方法的主循环结束后，增加"剩余量转新增排产"逻辑
   - 遍历已完成换活字块排产的SKU，计算 `窗口目标量 - 已排量 = 剩余量`
   - 如果剩余量 > 0，构建新增排产DTO加入待排列表

2. **`NewSpecProductionStrategy` 修改**：
   - 接收来自换活字块接力的SKU时，需要感知该SKU已在哪些机台排产（避免重复选机台）
   - 换模机台选择时排除已有换活字块排产的机台

3. **排产阶段衔接修改**：
   - 确认换活字块 → 新增排产的执行顺序和数据传递方式

### 边界场景

- 换活字块排完后剩余量极小（不足一班产），是否仍要增机台？→ 按现有新增排产逻辑的dayN模拟判断
- 换活字块机台与新增排产机台的收尾时间错开
- 换活字块SKU同时也是收尾SKU的场景

### 风险点

- 换活字块接力到新增排产后，SKU的目标量、窗口额度需要正确扣减，避免重复排产
- 需确认已排量（换活字块119条）是否已正确反映在日计划扣减中

## 问题二 & 三：非收尾SKU多机台尾量错开

### 改造目标

当同一个非收尾SKU在多台机台排产时，TAIL_MACHINE 应根据逐日计划量判断是否提前停止排产，避免在日计划已不欠产的班次继续排产。

### 现有逻辑影响点

| 类 / 方法 | 位置 | 说明 |
|-----------|------|------|
| `NewSpecProductionStrategy.adjustSameSkuMultiMachineEndingStagger()` | ~L2817 | 当前只处理收尾SKU |
| `NewSpecProductionStrategy` TAIL_MACHINE 计划量决策 | ~L1336 `resolveTailFillPlanQty()` | 按窗口累计追补，未按逐日截断 |
| dayN机台模拟 | ~L1700 | 模拟结果未反馈到计划量截断 |

### 设计思路

**方案A（推荐）：扩展尾量错开逻辑，覆盖非收尾SKU**

在 `adjustSameSkuMultiMachineEndingStagger()` 中，增加非收尾SKU的处理分支：
- 对于非收尾SKU的多机台排产，检查 TAIL_MACHINE 的最后排产班次对应的日期
- 如果该日期的累计排产量已满足当日日计划量（含前续借用），则清除 TAIL_MACHINE 在该班次的排产量

**方案B：在 TAIL_MACHINE 计划量决策时就做逐日截断**

在 `resolveTailFillPlanQty()` 中，不仅看窗口整体追补，还要逐日检查：
- 如果 TAIL_MACHINE 排到某一天时，该天的累计排产（所有机台合计）已满足日计划量，则截断不再排

**推荐方案A**，原因：
1. 方案A改动范围更小，在已有的 `adjustSameSkuMultiMachineEndingStagger()` 方法中扩展
2. 方案B需要修改计划量核心决策逻辑，影响面更大
3. 方案A的"事后调整"模式与现有收尾错峰逻辑一致，风格统一

### 详细实施步骤

1. **`adjustSameSkuMultiMachineEndingStagger()` 方法修改**：
   - 移除收尾判断的前置条件（`isEnd=1`），改为：同SKU多机台 && TAIL_MACHINE角色
   - 新增非收尾场景的尾量错开逻辑：
     - 收集同SKU所有机台的班次排产数据
     - 逐日累计排产量，与日计划量对比
     - 如果某一天累计排产已满足日计划量，则 TAIL_MACHINE 在该日及之后的班次排产量清零
     - 清除的量不需要转移到其他机台（非收尾SKU不需要"承接"）

2. **日计划扣减同步调整**：
   - 清除 TAIL_MACHINE 尾部班次后，重新计算该SKU的总排产量
   - 更新日计划滚动台账中的 `actualQty`、`scheduledQty` 等字段

3. **满班补齐超排逻辑适配**：
   - 清除的班次原本可能有满班补齐超排，清除后需要同步移除超排量

### 边界场景

- 两台机台在同一天收尾且都有超排 → 只需清除 TAIL_MACHINE 的超排
- TAIL_MACHINE 排产量清除后，该机台后续衔接的下一个SKU换模时间需要前移
- 非收尾SKU只有一台机台时不需要错峰
- 日计划量为0的日期（无需排产的日期）

### 风险点

- 清除尾部班次后，机台收尾时间提前，可能影响后续SKU的换模排队
- 需确认日计划台账的 `futureBorrowQty` 计算不受影响

## 问题三额外：换模时间中班而非早班

### 分析结论

**这是换模均衡策略的正常行为**。早班换模上限8台已被更高优先级的SKU占满，3302002534（rank=7）的K1002和K2027被顺延到中班14:00。

### 是否需要修改

这取决于业务要求：
- 如果业务上认为换模应尽量安排在早班 → 需要调整早班换模上限或优化SKU排序
- 如果中班换模本身不影响排程效率 → 不需要修改

**建议不修改**，中班换模是均衡策略的合理结果。如果确实需要调整，可以：
1. 提高早班换模上限（从8台→9台或10台）
2. 或者优化3302002534的排产优先级
