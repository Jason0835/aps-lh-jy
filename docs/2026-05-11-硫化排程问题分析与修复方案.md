## 二、问题详细分析

### 2.1 问题1：K2024 换活字块排错物料

**涉及文件**：
- `aps-lh/.../engine/strategy/impl/TypeBlockProductionStrategy.java`
- `aps-lh/.../engine/strategy/impl/DefaultSkuPriorityStrategy.java`

**现有逻辑链路**：

1. `ContinuousProductionHandler.doHandle()` 调用 `typeBlockProductionStrategy.scheduleTypeBlockChange(context)`（第51行）
2. `TypeBlockProductionStrategy.scheduleTypeBlockChange()` 收集收尾机台，按 estimatedEndTime 排序（第106-109行）
3. 对每个收尾机台，分两层匹配候选SKU：

**第一层**（第166-168行）：
```java
List<SkuScheduleDTO> priorityOneCandidates =
    filterSameEmbryoDescAndMainPatternCandidates(context, machine);
```
条件：同胎胚描述（embryoDesc） + 同主花纹（mainPattern） + 机台硬性匹配

**第二层**（第169-170行）：
```java
List<SkuScheduleDTO> priorityTwoCandidates =
    !strictPriorityOneAfterFirstHit && CollectionUtils.isEmpty(priorityOneCandidates)
        ? filterSameSpecCandidates(context, machine) : new ArrayList<>(0);
```
条件：同规格（spec） + 机台硬性匹配，且仅在未中过第一层时才启用

**选择逻辑**（第171-173行）：
```java
SkuScheduleDTO typeBlockSku = !CollectionUtils.isEmpty(priorityOneCandidates)
    ? selectPreferredSkuFromCandidates(priorityOneCandidates)
    : selectPreferredSkuFromCandidates(priorityTwoCandidates);
```

`selectPreferredSkuFromCandidates` 直接取 `candidates.get(0)`（第498-503行），即候选列表的第一个。

**根因分析**：

K2024 当前在机物料与 3302002654 和 3302002546 的判定：
- 如果两者都满足第一层条件（同胎胚描述 + 同主花纹），则都进入 priorityOneCandidates
- 候选列表保持 newSpecSkuList 的顺序
- 如果 3302002546 在排序列表中位于 3302002654 之前，则 3302002546 被选中
- **核心问题：候选SKU内部没有做二次排序，直接取 first()**

**具体原因**：
1. `selectPreferredSkuFromCandidates` 只取 `candidates.get(0)`，未在候选集内部按业务优先级再次排序
2. SKU 排序顺序本身可能不正确（见问题4），导致 3302002546 排在 3302002654 前面
3. 换活字块场景下，应在同层候选中优先选择收尾SKU、延误天数多、高优先级的SKU

### 2.2 问题2：物料 3302002546 应上 K1105

**涉及文件**：
- `aps-lh/.../handler/ScheduleAdjustHandler.java`（classifyContinuousAndNewSkus 方法）
- `aps-lh/.../handler/DataInitHandler.java`（buildStandardDataObjects 方法）
- `aps-lh/.../service/impl/RollingScheduleHandoffService.java`

**现有逻辑链路**：

1. `DataInitHandler.buildStandardDataObjects()` 初始化机台状态，从 `machineOnlineInfoMap` 获取 MES 在机物料（第202-213行）
2. `ScheduleAdjustHandler.classifyContinuousAndNewSkus()` 区分续作和新增SKU（第919-958行）：
   - 遍历 `machineOnlineInfoMap`，获取每台机台的 MES 在机物料编码
   - 调用 `resolveContinuousMaterialCode()` 决定续作物料：
     - 优先取滚动衔接后的机台当前物料（已继承且未收尾）
     - 否则取 MES 在线物料编码
   - 将匹配到的 SKU 标记为续作类型，设置 `continuousMachineCode`

**根因分析**：

物料 3302002546 应作为续作SKU上 K1105，但实际是 3302001575 上了 K1105，说明：

**场景A：MES在机数据问题**
- K1105 的 MES 在机信息（`machineOnlineInfoMap`）中记录的是物料 3302001575
- 如果数据库/MES上报的数据就是 3302001575，代码逻辑会正确地将 3302001575 分配到 K1105
- **这可能是上游数据问题，不是代码逻辑问题**

**场景B：滚动衔接覆盖了MES在机数据**
- `resolveContinuousMaterialCode` 优先使用滚动衔接后的机台当前物料（第1115-1117行）
- 如果上轮排程 K1105 最后一个排产结果物料是 3302001575 且标记未收尾，滚动衔接会将其继承为当前物料
- 然后 3302001575 被标记为续作，占用了 K1105

**场景C：SKU多机台副本分配问题**
- `assignContinuousSku` 方法（第989-1023行）支持同物料多机台续作
- 如果 3302002546 在 MES 快照中被其他机台先消费，K1105 可能分配到其他物料

**排查建议**：
需确认日志中：K1105 的 MES 在机物料、滚动衔接继承的 K1105 当前物料、以及 K1105 的 `machineOnlineInfoMap` 数据。

### 2.3 问题3：特殊材料 3302001538 应上 K1001

**涉及文件**：
- `aps-lh/.../engine/strategy/impl/DefaultMachineMatchStrategy.java`
- `aps-lh/.../util/LhSpecialMaterialUtil.java`
- `aps-lh/.../util/LhMachineHardMatchUtil.java`
- `aps-lh/.../engine/strategy/impl/DefaultSkuPriorityStrategy.java`

**现有逻辑链路**：

1. **特殊材料判定**（`LhSpecialMaterialUtil.resolveMatchResult`）：
   - 先按物料编码匹配 `specialMaterialCategoryByMaterialCode` Map
   - 未命中则按结构名称匹配 `specialMaterialCategoryByStructureName` Map
   - 返回分类列表（如 WIDE_BASE_195、CHIP_TIRE 等）

2. **机台过滤**（`DefaultMachineMatchStrategy.resolveSpecialSupportAvailabilityReason`，第235-259行）：
   - 对每个特殊分类，检查机台是否支持（`LhMachineHardMatchUtil.isCategorySupported`）
   - 不支持则返回对应不可用原因，该机台被过滤

3. **机台排序**（`buildMachineComparator`，第410-465行）：
   - 对于特殊材料SKU，`compareNormalMachinePriority` 返回 0（跳过，第478-480行）
   - 后续排序维度：endingTime → specExactMatch → proSizeExactMatch → inchDistance → capsuleAffinity → embryoShareCount → machineOrder → machineCode

**根因分析**：

3302001538 是特殊材料，需要上支持该特殊分类的机台 K1001。但实际是 3302002534（非特殊材料）上了 K1001，说明：

**根因1：SKU排序顺序导致非特殊SKU先抢占机台**
- 3302002534 在 newSpecSkuList 中排在 3302001538 之前
- 3302002534 先执行选机流程，且 3302002534 是非特殊材料，其 `buildMachineComparator` 中：
  - `compareNormalMachinePriority` 将普通机台（`isNormalMachine=true`）排在前面（score=0）
  - 支持特殊材料的 K1001 因 `isNormalMachine=false`（score=1），排名靠后
  - **但是**：如果其他普通候选机台因为各种原因被过滤（寸口、模套、模具占用等），K1001 可能成为 3302002534 的首选机台
- 3302002534 占用了 K1001 后，3302001538 排到时 K1001 的 estimatedEndTime 已被推后
- 3302001538 可能因为其他候选机台不支持其特殊分类而排程失败

**根因2：特殊材料机台匹配缺少专用优先级**
- 对于特殊材料SKU，`compareNormalMachinePriority` 被跳过（返回0），不做区分
- 没有将"支持该特殊分类的机台"排在更前面，也没有将"完全兼容的特殊机台"与"仅因被过滤剩下而成为候选的普通机台"区分

**根因3：SKU排序未考虑特殊材料应优先**
- 在开产模式下，特殊材料会被加上 `OPEN_PRODUCTION_SPECIAL_MATERIAL_PENALTY` 靠后分（第212-217行）
- 这意味着特殊材料在开产模式下反而排在更后面
- 如果没有开产模式，特殊材料没有特殊排序优势

### 2.4 问题4：SKU排序顺序不正确

**涉及文件**：
- `aps-lh/.../engine/strategy/impl/DefaultSkuPriorityStrategy.java`

**期望顺序**：3302002654 → 3302002546 → 3302002177 → 3302001513

**现有排序规则**（`buildSkuComparator`，第93-115行）：

| 排序维度 | 排序方向 | 说明 |
|---------|---------|------|
| deliveryLocked | 锁定优先 | 有发货要求排前 |
| delayDays 是否已知 | 已知优先 | delayDays >= 0 排前 |
| delayDays 值 | 越大越优先 | 延误天数多的排前 |
| structureAllEndingPriority | 命中优先 | 结构全收尾的排前 |
| endingDaysRemaining 是否已知 | 已知优先 | |
| endingDaysRemaining 值 | 越大越优先 | 收尾越晚越先上机 |
| highPriorityPendingQty | 越多越优先 | 高优先级待排量 |
| cycleProductionPendingQty | 越多越优先 | 周期排产待排量 |
| midPriorityPendingQty | 越多越优先 | 中优先级待排量 |
| conventionProductionPendingQty | 越多越优先 | 常规储备待排量 |
| openProductionLateScore | 越小越优先 | 开产靠后分 |
| materialCode | 字典序 | 兜底 |

**可能原因分析**：

1. **delayDays 字段影响**：如果某个SKU的 `beginDay`（月计划起产日）配置不同，会导致 delayDays 差异，改变排序

2. **供应链优先级字段影响**（第108-111行）：
   ```java
   .thenComparingInt((SkuScheduleDTO s) -> -s.getHighPriorityPendingQty())
   .thenComparingInt((SkuScheduleDTO s) -> -s.getCycleProductionPendingQty())
   .thenComparingInt((SkuScheduleDTO s) -> -s.getMidPriorityPendingQty())
   .thenComparingInt((SkuScheduleDTO s) -> -s.getConventionProductionPendingQty())
   ```
   月计划中的 `HEIGHT_PRODUCTION_QTY`、`CYCLE_PRODUCTION_QTY`、`MID_PRODUCTION_QTY`、`CONVENTION_PRODUCTION_QTY` 字段值会影响排序。如果这些字段存在非零值且与期望不符，会导致排序偏移。

3. **开产模式靠后分**：如果是开产模式：
   - 特殊材料加 100 分（`OPEN_PRODUCTION_SPECIAL_MATERIAL_PENALTY`）
   - 雪地胎加 200 分
   - 不同英寸加 50 分
   - 这些惩罚分会把某些SKU推后

4. **结构全收尾优先级**：`buildStructurePriorityMap` 检查结构内是否所有SKU都已收尾：
   - 如果结构A全部收尾且 remainingDays 在阈值内，该结构内SKU获得优先
   - 这可能导致某些SKU被意外提前

5. **materialCode 字典序兜底**：3302001513 < 3302002177 < 3302002546 < 3302002654（字典序）
   - 如果前面所有维度都相同，按此兜底顺序排列
   - 期望顺序（2654 → 2546 → 2177 → 1513）恰好是字典序的**逆序**
   - 这说明期望顺序被前面的维度打乱了

**核心矛盾**：期望顺序 2654→2546→2177→1513 与 materialCode 字典序恰好相反，说明期望顺序应该由业务字段（delayDays、待排量、优先级等）驱动，而不是由字典序兜底。

### 2.5 问题5：多机台分配时班次未满产

**涉及文件**：
- `aps-lh/.../engine/strategy/impl/NewSpecProductionStrategy.java`（scheduleNewSpecs 方法）
- `aps-lh/.../component/TargetScheduleQtyResolver.java`（refineTargetQtyByMachineCapacity 方法）
- `aps-lh/.../engine/strategy/impl/ContinuousProductionStrategy.java`（distributeToShifts 方法）

**现有逻辑链路**：

1. **多机台拆量循环**（`NewSpecProductionStrategy.scheduleNewSpecs`，第228-407行）：
```java
while (true) {
    // 选择候选机台
    MachineScheduleDTO candidateMachine = selectCandidateMachine(...);

    // 收敛目标量：满排模式 = min(剩余量, 机台实际窗口产能)，非满排模式 = 剩余量
    int refinedTargetQty = getTargetScheduleQtyResolver()
        .refineTargetQtyByMachineCapacity(context, sku, candidateMachine,
                                          mouldChangeStartTime, firstProductionStartTime, shifts);

    sku.setTargetScheduleQty(refinedTargetQty);
    LhScheduleResult result = buildNewSpecScheduleResult(...);

    int machineScheduledQty = result.getDailyPlanQty();
    remainingQty = Math.max(0, remainingQty - machineScheduledQty);

    if (remainingQty <= 0 || !needMoreMachine(sku)) {
        break;
    }
    sku.setTargetScheduleQty(remainingQty); // 下台机台只排剩余量
}
```

2. **目标量收敛**（`TargetScheduleQtyResolver.refineTargetQtyByMachineCapacity`，第87-105行）：
```java
public int refineTargetQtyByMachineCapacity(...) {
    int currentTargetQty = sku.resolveTargetScheduleQty();
    if (currentTargetQty <= 0 || !isFullCapacityMode(context)) {
        return Math.max(currentTargetQty, 0); // 非满产模式直接返回剩余量
    }
    int actualCapacityQty = resolveActualWindowCapacity(...);
    return Math.min(currentTargetQty, actualCapacityQty); // 满产模式取 min(剩余, 产能)
}
```

3. **班次分配**（`distributeToShifts`，第830-912行）：
```java
int shiftQty = ShiftCapacityResolverUtil.normalizeAllocatedShiftQty(
    Math.min(remaining, shiftMaxQty), shiftMaxQty, mouldQty);
// remaining 是 SKU 的剩余目标量，不是机台班次的满产量
```

**根因分析**：

在满排模式下（`isFullCapacityMode=true`），问题的核心在于 `refineTargetQtyByMachineCapacity` 返回 `Math.min(currentTargetQty, actualCapacityQty)`：

- 假设 SKU 总目标量 = 300
- 机台A 实际窗口产能 = 200，则 refinedTargetQty = min(300, 200) = 200 → 机台A 满产
- 剩余量 = 300 - 200 = 100
- 机台B 实际窗口产能 = 200，则 refinedTargetQty = min(100, 200) = 100 → **机台B 班次未满产**

**最后一台机台的班次**会因为剩余量不足而无法满产。虽然这符合"不超额排产"的业务语义，但用户期望在多机台场景下，每台机台都应满产。

在非满排模式下（默认），`refineTargetQtyByMachineCapacity` 直接返回 `currentTargetQty`（即剩余量），情况更糟：每台机台都只排到剩余量，班次可能更不满。

**核心矛盾**：
- 当前逻辑：多机台分配时，"剩余量递减"导致后面的机台排不满
- 期望逻辑：每台机台应按其产能满排，超出的部分通过 `shiftFillOverQty` 机制冲抵后续日计划

### 3.2 问题4修复：SKU排序顺序修正

**改造目标**：确保 SKU 按正确的业务优先级排序。

**改造文件**：
- `DefaultSkuPriorityStrategy.java` - `buildSkuComparator` 方法
- `DefaultSkuPriorityStrategy.java` - `buildNewSpecComparator` 方法

**方案**：

#### 方案A：增强换活字块候选选择（小改动，治标）

在 `TypeBlockProductionStrategy` 中修改候选SKU选择逻辑，不直接取 `candidates.get(0)`，而是在候选集内按业务优先级二次排序：

```java
// 修改 selectPreferredSkuFromCandidates 方法
private SkuScheduleDTO selectPreferredSkuFromCandidates(List<SkuScheduleDTO> candidates) {
    if (CollectionUtils.isEmpty(candidates)) {
        return null;
    }
    if (candidates.size() == 1) {
        return candidates.get(0);
    }
    // 在多候选时，优先选择收尾SKU、高优先级的SKU
    candidates.sort(Comparator
        .comparingInt((SkuScheduleDTO s) ->
            SkuTagEnum.ENDING.getCode().equals(s.getSkuTag()) ? 0 : 1)
        .thenComparingInt(s -> s.getScheduleOrder() > 0 ? s.getScheduleOrder() : Integer.MAX_VALUE));
    return candidates.get(0);
}
```

#### 方案B：修正 SKU 排序比较器（大改动，治本）

需要结合具体数据分析 delayDays、待排量等字段为何导致排序不一致，调整比较器逻辑。建议：

1. 添加"特殊材料SKU优先"维度（放在供应链优先级之后、开产靠后分之前）：
```java
// 新增：特殊材料优先于非特殊材料（同条件下）
.thenComparingInt((SkuScheduleDTO s) ->
    isSpecialMaterial(context, s) ? 0 : 1)
```

2. 检查 `resolveOpenProductionLateScore` 对特殊材料的影响：
   - 如果当前是开产模式，特殊材料加 100 分惩罚会导致其排后
   - 如果期望特殊材料应优先，需要调整此逻辑

3. 如果没有开产模式，检查供应链优先级四个字段的数据是否正确

**推荐方案**：
- 先实施方案A（改 `TypeBlockProductionStrategy.selectPreferredSkuFromCandidates`），解决换活字块候选选择问题
- 根据数据库实际数据验证 SKU 排序字段，确定方案B是否需要调整

### 3.3 问题1修复：换活字块候选增强

**改造目标**：确保 K2024 换活字块时优先选择 3302002654 而非 3302002546。

**改造文件**：
- `TypeBlockProductionStrategy.java` - `selectPreferredSkuFromCandidates` 方法（第498-503行）

**设计思路**：

当前 `selectPreferredSkuFromCandidates` 只返回 `candidates.get(0)`，未做二次筛选。应在候选集中增加业务优先级排序：
1. 收尾SKU优先（`skuTag == ENDING`）
2. 按 `scheduleOrder` 升序（与 SKU 优先级排序一致）
3. 高优先级待排量多的优先
4. 延误天数多的优先

**实施步骤**：

1. 修改 `selectPreferredSkuFromCandidates` 方法
2. 在同一层候选人中增加多维度排序
3. 更新关联的 `appendSpecifyTypeBlockResult` 调用路径（该方法也使用了相同的候选筛选逻辑）

**边界场景**：
- 定点物料换活字块（`appendSpecifyTypeBlockResult`）：定点物料始终优先，不受影响
- 第一层命中多候选：按新排序选择最优
- 第二层命中多候选：同第一层逻辑

### 3.4 问题3修复：特殊材料机台匹配优先级增强

**改造目标**：确保特殊材料 SKU 优先被分配到支持该特殊分类的机台。

**改造文件**：
- `DefaultMachineMatchStrategy.java` - `buildMachineComparator` 方法
- `DefaultSkuPriorityStrategy.java` - `buildSkuComparator` 方法（配合问题4）

**方案**：

#### 方案3A：机台排序增加特殊材料专用优先级

在 `buildMachineComparator` 中，对特殊材料 SKU 增加"支持该分类的机台优先"维度：

```java
// 在 compareNormalMachinePriority 之后，compareEndingTime 之前插入
compareResult = compareSpecialMaterialSupportPriority(matchResult, left, right);
if (compareResult != 0) {
    return compareResult;
}
```

```java
private int compareSpecialMaterialSupportPriority(SpecialMaterialMatchResult matchResult,
                                                   MachineScheduleDTO left,
                                                   MachineScheduleDTO right) {
    if (Objects.isNull(matchResult) || !matchResult.isSpecial()) {
        return 0;
    }
    int leftSupport = LhMachineHardMatchUtil.isSpecialMaterialSupported(matchResult, left) ? 0 : 1;
    int rightSupport = LhMachineHardMatchUtil.isSpecialMaterialSupported(matchResult, right) ? 0 : 1;
    return Integer.compare(leftSupport, rightSupport);
}
```

#### 方案3B：SKU排序中特殊材料提前

修改 `DefaultSkuPriorityStrategy.buildNewSpecComparator`，将特殊材料 SKU 排在前面：

```java

    return Comparator
        .comparingInt((SkuScheduleDTO sku) -> LhSpecifyMachineUtil.hasLimitSpecifyMachine(
                context, sku.getMaterialCode()) ? 0 : 1)
        .thenComparingInt(sku -> sku.isTrial() ? 0 : 1)

}
```

### 3.6 问题5修复：班次满产

**改造目标**：在一台SKU分配到多机台时，每台机台的班次应按产能满排（满排模式下）。

**改造文件**：
- `TargetScheduleQtyResolver.java` - `refineTargetQtyByMachineCapacity` 方法

**设计思路**：

当前逻辑在满排模式下返回 `Math.min(currentTargetQty, actualCapacityQty)`，导致最后一台机台因剩余量不足而无法满产。

修改为：在满排模式下，每台机台都按其实际窗口产能满排（返回 `actualCapacityQty`），超出 SKU 总需求的部分通过现有的 `shiftFillOverQty` 机制处理。

**实施步骤**：

1. 修改 `refineTargetQtyByMachineCapacity`：
```java
public int refineTargetQtyByMachineCapacity(...) {
    int currentTargetQty = sku.resolveTargetScheduleQty();
    if (currentTargetQty <= 0 || !isFullCapacityMode(context)) {
        return Math.max(currentTargetQty, 0);
    }
    int actualCapacityQty = resolveActualWindowCapacity(...);
    if (actualCapacityQty <= 0) {
        return 0;
    }
    // 满排模式：每台机台排满，超出部分由 shiftFillOverQty 机制处理
    return actualCapacityQty;
}
```

2. 确保 `applyBlockToDailyQuota` 中的满班超排处理正确工作（已存在，无需修改）

3. 调整多机台循环中的 `needMoreMachine` 判断，因为目标量改为按产能后，单台机台就可能排超：
```java
// 满排模式下，剩余量递减逻辑不变，但 refinedTargetQty = actualCapacityQty
// 排超的部分会自动计入 shiftFillOverQty
```

**边界场景**：
- 非满排模式：行为不变
- 单机台场景：行为不变（actualCapacityQty >= currentTargetQty 时不变）
- 多机台场景：最后一台机台也满排，超出部分走满班超排

**风险点**：
- 可能造成超出月计划余量的排产，需确保 `shiftFillOverQty` 追踪和后续日计划冲抵正确
- 需要在多机台循环中正确处理 `remainingQty < 0` 的情况
