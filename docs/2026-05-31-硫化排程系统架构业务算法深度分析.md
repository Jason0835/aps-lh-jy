# 硫化排程系统架构、业务与算法深度分析

分析日期：2026-05-31

入口方法：`com.zlt.aps.lh.controller.LhScheduleResultController#executeSchedule`

分析范围：`aps-lh` 模块当前代码，重点覆盖 Controller、Service、Engine、Handler、Strategy、Context、数据初始化、结果校验与持久化链路。

说明：本报告是只读代码审计结论，未启动服务、未触发真实排程批次。涉及业务正确性的高风险判断，后续仍建议用真实批次和数据库结果闭环验证。

---

## 1. 明确结论

当前硫化排程系统的总体架构方向是合理的：HTTP 入口较薄，Service 负责构建排程上下文，执行器使用装饰器增强日志和性能监控，模板方法固定 `S4.1 -> S4.6` 主流程，Handler 做阶段编排，Strategy 承载不同排程规则，最终通过事务服务原子替换目标日结果。

但当前系统已经进入“业务规则高度集中、上下文状态高度共享、核心策略类过大”的阶段。主要风险不在于入口链路，而在于以下三点：

1. `LhScheduleContext` 是全流程可变数据总线，多个 Handler/Strategy 直接读写同一批 Map/List，状态边界弱，后续需求频繁变更时容易出现链路不一致。
2. `NewSpecProductionStrategy`、`ContinuousProductionStrategy`、`TypeBlockProductionStrategy`、`ResultValidationHandler` 等类体量过大，单类承载选机、窗口、账本、库存、机台状态、日志、回流等多种职责，维护和回归成本偏高。
3. 算法本质是“分阶段贪心 + 局部规则修正 + 受限 DFS 评估”，可解释性强、工程落地稳，但对 SKU 顺序和机台顺序敏感，局部搜索目前更多是辅助评估，尚未形成全局或半全局优化能力。

推荐优化方向不是大范围重构，也不建议引入新求解器。应按“先补契约和校验，再小步拆分超大策略，最后让局部搜索真正参与决策”的顺序推进。

## 2. 当前主调用链路

### 2.1 入口链路

当前真实代码链路如下：

```text
POST /lhScheduleResult/execute
  -> LhScheduleResultController.executeSchedule()
     -> LhScheduleServiceImpl.executeSchedule()
        -> buildContext()
        -> IScheduleExecutor.execute()
           -> PerformanceMonitorDecorator
              -> LoggingScheduleDecorator
                 -> DefaultScheduleExecutor
                    -> AbsLhScheduleTemplate.execute()
                       -> S4.1 PreValidationHandler
                       -> S4.2 DataInitHandler
                       -> S4.3 ScheduleAdjustHandler
                       -> S4.4 ContinuousProductionHandler
                       -> S4.5 NewProductionHandler
                       -> S4.6 ResultValidationHandler
                          -> SchedulePersistenceService.replaceScheduleAtomically()
                          -> ScheduleEventPublisher.publish()
```

核心证据：

- `LhScheduleResultController#executeSchedule` 使用 `@DistributedLock`，锁 key 为 `APS:LH:SCHEDULE:LOCK:{factoryCode}:{scheduleDate}`，Controller 只记录日志并委托 Service。
- `LhScheduleServiceImpl#executeSchedule` 构建 `LhScheduleContext` 后委托 `scheduleExecutor.execute(context)`。
- `ScheduleExecutorConfig` 组装装饰器链：`PerformanceMonitor -> Logging -> DefaultScheduleExecutor`。
- `DefaultScheduleExecutor` 只委托 `AbsLhScheduleTemplate#execute`。
- `AbsLhScheduleTemplate#execute` 是 `final` 方法，固定六阶段执行顺序。
- `ResultValidationHandler` 最终调用 `SchedulePersistenceService#replaceScheduleAtomically` 并发布完成事件。

### 2.2 需要纠正的旧认知

项目中存在 `ScheduleExecutionGuard`，其实现是 Redis `setIfAbsent` + Lua 释放锁，但当前代码未检索到任何调用点。当前实际生效的并发控制是 Controller 上的 `@DistributedLock` 和 `DistributedLockAspect`。

这意味着：

- 不能再把 `ScheduleExecutionGuard` 视为当前主链路的一部分。
- 如果后续维护文档、代码注释或方案仍写“Service 层获取 ScheduleExecutionGuard 锁”，需要同步修正。
- 保留两套锁组件会造成认知成本，建议明确二选一：当前继续使用 `@DistributedLock`，则标记 `ScheduleExecutionGuard` 为待清理或迁移；若改回 Guard，则应统一入口并删除 Controller 注解锁。

## 3. 代码架构层面分析

### 3.1 MVC 分层评价

当前分层总体清晰：

| 层级 | 代表类 | 当前职责 | 评价 |
| --- | --- | --- | --- |
| Controller | `LhScheduleResultController` | 接收排程/发布请求、分布式锁、请求日志 | 较薄，职责基本合理 |
| Service | `LhScheduleServiceImpl` | 构建上下文、调用执行器、发布结果状态更新 | 基本合理，但异常转响应集中在这里 |
| Engine | `IScheduleExecutor`、装饰器、模板 | 固定执行骨架、日志/性能增强 | 设计方向正确 |
| Handler | `PreValidationHandler` 等 | 六阶段流程编排 | 阶段边界清楚 |
| Strategy | `ContinuousProductionStrategy`、`NewSpecProductionStrategy` 等 | 具体业务算法和规则 | 规则沉淀充分，但类过大 |
| Context | `LhScheduleContext` | 全流程数据总线 | 能支撑复杂业务，但耦合风险高 |
| Persistence | `SchedulePersistenceService` | 原子替换目标日结果 | 事务边界明确 |

结论：MVC 与引擎分层本身合理，主要问题是 Strategy 层内部职责继续膨胀，Context 成为隐式依赖中心。

### 3.2 引擎设计模式

当前系统已使用多个适合排程领域的设计模式：

| 设计模式 | 使用位置 | 作用 | 当前评价 |
| --- | --- | --- | --- |
| 模板方法 | `AbsLhScheduleTemplate#execute` | 固定 S4.1-S4.6 主流程 | 合理，避免主流程被策略随意改写 |
| 策略模式 | `IMachineMatchStrategy`、`ISkuPriorityStrategy`、`IProductionStrategy` 等 | 抽象不同排程规则 | 合理，但部分策略实现过大 |
| 简单工厂/注册表 | `ScheduleStrategyFactory` | 注册生产策略和通用策略 | 可用，但对多实现扩展不够柔性 |
| 装饰器 | `LoggingScheduleDecorator`、`PerformanceMonitorDecorator` | 非侵入增强执行日志和性能统计 | 合理 |
| 责任链 | `DataValidationChain` + `IDataValidator` | 基础数据校验分组执行 | 合理，便于新增校验器 |
| 观察者 | `ScheduleEventPublisher`、多个 Listener | 排程完成/发布后扩展后置动作 | 合理，但异常处理语义需要明确 |
| 快照 | `LhScheduleConfig` | 将参数解析为本次排程配置快照 | 合理，降低运行中参数变更影响 |

### 3.3 模块耦合度和内聚性

#### 3.3.1 高内聚的部分

以下模块职责较清晰：

- `PreValidationHandler`：发布状态、跨月、批次号。
- `DataValidationChain`：基础数据校验编排。
- `SchedulePersistenceService`：保存前审计字段、班次收尾标记、旧数据删除、新数据批量插入。
- `ScheduleExecutorConfig` 与装饰器：执行链增强。
- `LhScheduleConfigResolver` 与 `LhScheduleConfig`：参数解析与快照。

#### 3.3.2 内聚不足的部分

核心策略类已经超过“单一策略”的合理边界：

| 类 | 行数 | 主要职责堆叠 |
| --- | ---: | --- |
| `NewSpecProductionStrategy` | 5447 | 新增 SKU 遍历、机台候选、局部搜索、换模、首检、产能、班次分配、账本扣减、库存裁剪、结果构建、机台状态更新、未排合并 |
| `ContinuousProductionStrategy` | 4028 | 续作、收尾、同 SKU 多机台、副本共享账本、换活字块前置关系、降模、补偿 SKU、库存裁剪、机台状态回写 |
| `TypeBlockProductionStrategy` | 2988 | 换活字块候选机台、定点物料预留、兜底机台、切换窗口、日计划账本回裁、结果回流 |
| `ResultValidationHandler` | 1564 | 结果必填校验、数量策略校验、换模计划、清洗阈值、工单号、排序、日志、示方保护、保存 |
| `ScheduleAdjustHandler` | 1469 | 欠超产、月计划展开、日计划账本、库存分摊、收尾判定、续作/新增分类 |
| `LhBaseDataServiceImpl` | 1489 | 25+ 类基础数据加载、并发任务、依赖任务、前批次数据、历史数据 |

这些类的共同问题是：业务含义强，但修改点过多。后续某个小规则变更，例如“试制 SKU 的机台优先级”或“续作非收尾挤量”，可能同时牵涉排序、选机、账本、结果校验和日志。

### 3.4 可扩展性和可维护性

#### 优点

1. 主流程固定，新增阶段不会轻易破坏 S4.1-S4.6。
2. 策略接口较完整，至少从命名上已覆盖机台匹配、SKU 排序、换模、首检、产能、试制、停产、收尾等关键领域。
3. 配置快照集中在 `LhScheduleConfig`，开关型规则可通过参数控制。
4. 数据校验链支持新增 Validator。
5. 持久化服务使用事务原子替换，减少“删旧成功、写新失败”的数据库风险。

#### 风险

1. `LhScheduleContext` 的字段过多，且多数是可变集合，调用方很难通过类型知道某个字段在哪个阶段有效、由谁负责维护。
2. `scheduleResultSourceSkuMap` 使用 `IdentityHashMap` 按对象身份绑定结果与 SKU，这能解决同物料多副本串账问题，但也要求对象复制、回流、分组时必须非常谨慎。
3. 续作、换活字块、新增之间存在大量共享状态，例如 `newSpecSkuList`、`structureSkuMap`、`dailyPlanQuotaMap`、`machineScheduleMap`、`skuShiftFillOverQtyMap`。任何一处忘记同步，后续阶段都可能拿到过期状态。
4. `ScheduleStrategyFactory` 对通用策略提供固定 getter，新增同接口多实现时需要额外处理 Spring 注入歧义，不如生产策略注册表灵活。
5. 观察者当前同步执行并吞掉 listener 异常。排程主结果不受影响，但精度计划回填、MES 通知等后置动作失败可能只留日志，业务上需要明确是否允许弱一致。

### 3.5 可配置化评价

当前可配置能力较强，典型参数包括：

- 排程天数、满产排程、强制重排、定点机台规则。
- 局部搜索开关、搜索深度、时间预算、候选窗口。
- 新增/续作欠产前瞻天数。
- 月末余量收尾、结构收尾天数。
- 首检、换模、开停产、干冰/喷砂、保养等规则参数。

主要问题：

1. `LhScheduleConfigResolver` 对参数解析失败多采用 `warn + 默认值`，这对生产稳定性友好，但对排程准确性有风险。若参数配置错误，系统可能静默使用默认值，排程结果与业务预期不一致。
2. 建议增加“严格参数校验模式”。核心参数解析失败时，在 S4.2 校验阶段直接阻断，而不是继续默认值执行。
3. 配置项数量已经较多，建议输出“本次排程配置快照日志”或落库到过程日志，便于复盘某批次到底用了哪些开关和值。

### 3.6 异常处理评价

当前异常处理有三层：

1. Controller 分布式锁异常由 `LhScheduleExceptionHandler` 转为业务响应。
2. `LhScheduleServiceImpl` 捕获 `ScheduleException` 和普通 `Exception`，转为失败响应。
3. `AbsScheduleStepHandler` 默认将步骤异常写入 context 中断；`ResultValidationHandler#shouldPropagateException` 返回 `true`，保存前后关键失败会向上抛。

优点：用户侧可获得失败响应，S4.6 保存失败不会被普通中断吞掉。

风险：

1. 普通 Handler 默认中断不抛出，适合业务校验失败，但不适合系统异常或编程错误。建议区分“可预期业务中断”和“系统异常”。
2. `ScheduleEventPublisher` 捕获 Listener 异常后只打日志。对 MES 通知可接受，对精度计划回填是否可接受需业务确认。
3. `MesNotifyListener` 目前只是日志和 TODO，发布成功并不代表真正通知 MES。
4. 代码中仍存在 `SimpleDateFormat` 使用点，例如 `OrderNoGenerator`、`ResultValidationHandler`、`DefaultFirstInspectionBalanceStrategy`、`LhScheduleTimeUtil`。这与项目规范“禁止 SimpleDateFormat”冲突，建议后续小步替换为 Hutool 或 `java.time`，但需要保证工单号格式不变。

## 4. 业务逻辑层面分析

### 4.1 核心业务流程

#### S4.1 前置校验

职责：

- 校验目标日是否已有已发布 MES 排程结果。
- 校验排程窗口是否跨月。
- 生成批次号。

评价：

- 校验前置合理，避免覆盖已发布结果。
- S4.6 持久化前又二次校验发布状态，能防止计算期间目标日被发布。
- 跨月直接拒绝符合当前月计划 dayN 字段结构，但如果业务后续要求跨月，需要先改月计划账本模型，不能只放开校验。

#### S4.2 数据初始化

职责：

- 解析班次。
- 并行加载月计划、工作日历、SKU 产能、机台、停机、模具、胎胚库存、完成量、MES 在机、定点机台、保养、历史结果等数据。
- 数据校验链校验基础数据。
- 初始化机台运行态和班次运行态。
- 非强制重排时执行滚动衔接。

评价：

- 并行加载适合当前数据规模，能降低入口耗时。
- `CallerRunsPolicy` 可以避免线程池拒绝导致数据丢失，但在高压下会把加载压力推回主线程，排程响应时间可能上升。
- 数据初始化职责过重，建议后续将“基础数据加载”和“运行态构建”拆开。

#### S4.3 排程调整与 SKU 归集

当前实现的关键业务：

- 月计划余量使用“月计划总量 - 已完成量”计算，不再读取月余量表作为兜底。
- 日计划账本按窗口自然日读取月计划 dayN，禁止跨月。
- T 日晚班完成量会扣减日计划账本。
- T-1 欠产/超产通过 `carryForwardQtyMap` 和账本处理传导。
- 同胎胚库存按标准产能或日硫化量占比分摊。
- 目标量由 `TargetScheduleQtyResolver` 决定，支持满排模式、需求模式、严格目标量。
- 根据 MES 在机和滚动继承结果分类为续作和新增。

评价：

- 账本链路比简单的“按月计划余量排产”更贴近滚动排程真实场景。
- 续作多机台副本共享同一个 `dailyPlanQuotaMap`，能避免同物料多机台各自重复消费日计划。
- 风险是日计划账本是可变共享对象，后续 S4.4/S4.5/S4.6 都会消费它，必须依赖强校验防止重复扣减或漏扣。

潜在问题：

1. `calculateSurplusQty` 已明确不使用月余量表。如果月余量表仍承载人工调整、冻结量或外部修正，则当前口径可能忽略这些业务输入。
2. 同胎胚库存按整数比例分摊会产生尾差，多个 SKU 分摊总量可能小于胎胚库存。若业务要求库存充分利用，需要明确尾差归属规则。
3. SKU 产能缺失时构造 DTO 内有默认硫化时间逻辑。虽然前置 Validator 应该拦截产能缺失，但这类默认值仍可能掩盖数据问题，建议统一改为校验阻断或明确过程日志。

### 4.2 续作排产分析

S4.4 续作流程：

```text
SKU 排序
  -> 续作收尾排产
  -> 换活字块衔接
  -> 续作班次计划量分配
  -> 胎胚库存后置裁剪
  -> 续作降模
  -> 未满足目标的续作补偿 SKU 回流 S4.5
```

当前实现优点：

1. 续作优先于新增，符合硫化现场“先承接在机状态，再考虑换模新增”的业务顺序。
2. 单机台续作在满排模式下会按真实窗口产能修正目标量，多机台沿用共享账本和降模逻辑。
3. 续作非收尾可以为定点新增预留切换窗口，避免把后续必须换模的定点机台全部挤满。
4. 降模逻辑按业务日、dayN、前瞻天数、滚动差异选择保留机台和补量下机机台，能解释“为什么降这台、不降那台”。
5. 续作结果通过 `scheduleResultSourceSkuMap` 绑定来源 SKU，后续账本、库存和班次收尾标记能按来源对象分组。

潜在问题：

1. 续作“当前窗口无日计划量时释放机台给换模/新增”的规则需要业务确认。若存在停产收尾、库存收尾、强制追补等场景，这个释放可能过早。
2. 正式续作在满排场景下存在保留满班补齐、不做胎胚库存裁剪的规则。若业务上胎胚库存是硬约束，该口径需要调整；若满班补齐允许提前占用未来胎胚，则需要在报告和日志中明确。
3. `scheduleReduceMould` 同时负责降模、日账本同步、补偿 SKU、零计划结果、同班尾量归集、库存分摊和机台状态同步，职责过重，后续改动容易互相影响。

### 4.3 换活字块排产分析

`TypeBlockProductionStrategy#scheduleTypeBlockChange` 的核心逻辑：

1. 基于续作收尾回写后的 `estimatedEndTime` 找收尾机台。
2. 追加在机前规格兜底机台。
3. 每轮按触发来源、就绪时间、收尾时间排序机台。
4. 先处理定点物料；若定点物料更适合走新增换模主链，则预留给 S4.5。
5. 再按同胎胚、同模具、同主花纹等条件筛选换活字块候选 SKU。
6. 避开晚班不可换模、保养、停机等窗口。
7. 构建 `scheduleType=03`、`isTypeBlock=1` 的结果，并按日计划账本回裁。
8. 若单台产能不足，剩余量回流 S4.5 新增排产。

优点：

- 换活字块嵌在 S4.4，能自然承接续作收尾后的机台状态。
- 对定点机台和需要完整新增换模评估的 SKU 做了让渡，避免 S4.4 抢占 S4.5 主链。
- 结果仍进入统一 S4.6 换模计划和持久化链路。

潜在问题：

1. 策略中存在“收尾机台”和“兜底机台”两类触发来源，业务可解释，但“兜底机台”属于较强业务假设，建议保留明确过程日志和可开关能力。
2. 换活字块结果 `isChangeMould=1`，但业务语义又不是普通新增换模。当前通过 `isTypeBlock=1` 区分，下游 SQL、报表和 MES 接口必须统一识别，否则容易把换活字块误当普通换模。
3. `applyTypeBlockToDailyQuota` 对收尾严格截断、非收尾允许满班补齐，这与新增/续作逻辑相似，建议后续抽取同一套“日计划账本消费策略”，避免三处口径漂移。

### 4.4 新增换模排产分析

S4.5 新增流程复杂度最高，主要逻辑包括：

- 新增 SKU 排序。
- 获取候选机台。
- 单控/普通机台按试制、量试、小批量、正规分层处理。
- 局部搜索评估候选机台。
- 换模时间、首检、开产时间、保养、清洗、停机窗口综合计算。
- 按机台窗口计算班次产能。
- 根据日计划账本和目标量决定单机台排产量。
- 多机台动态拆量。
- 构建结果并回裁日计划账本。
- 后置库存裁剪、零计划结果处理、机台状态同步、未排合并。

当前实现优点：

1. 试制 SKU 严格使用单控机台；量试/小批量优先单控但允许普通机台；正规 SKU 优先普通机台，这与现场资源保护逻辑一致。
2. `resolveMachinePlanQty` 和动态多机台拆量逻辑能避免一个 SKU 在单机台上吃掉多日窗口，保留后续 dayN 节奏。
3. 新增非收尾允许满班补齐，并记录 `shiftFillOverQty`，这比直接截断班产更贴近机台生产实际。
4. 对换模、首检、开停产、清洗、保养都纳入班次分配，业务维度较完整。

潜在问题：

1. 局部搜索入口存在，但当前 S4.5 代码注释和实现表现为“只评估，不改写当前首选机台”的倾向。如果最终还是回到排序首台，局部搜索对结果的实际影响有限。
2. 新增主循环已经有代码 TODO 建议拆为“候选生成、窗口分配、结果构建、账本消费”四个阶段，说明当前维护成本已被代码本身承认。
3. 正式新增非试制场景可跳过胎胚库存裁剪以保留满班结果。该规则需要业务确认：如果胎胚库存是硬约束，应改为统一截断；如果不是硬约束，应在日志和报表中标明超库存来源。
4. 换模/首检分配失败后依赖 rollback 清理资源状态，后续改动必须覆盖失败分支测试，避免全局状态污染。

### 4.5 多机台计划量分配

当前多机台计划量分配的关键口径：

- 续作同物料多机台共享同一份 `dailyPlanQuotaMap`。
- 新增同 SKU 多机台通过动态目标量和日计划账本控制节奏。
- 多机台班次收尾标记在保存前统一计算：同 SKU 多机台场景中辅助机台最后一个有计划量班次也视为机台收尾。
- 降模时优先保留更符合胶囊使用、dayN 保障和产能口径的机台。

评价：

- 当前规则比简单均分更贴近现场：它不是平均分配，而是按候选机台、可用窗口、dayN、收尾和补满规则动态分配。
- 风险是“同 SKU 多机台”需要依赖 sourceSku 对象身份和共享账本。如果某个地方复制 DTO 后没有共享账本或没有写 `scheduleResultSourceSkuMap`，保存前分组、库存裁剪、日账本校验都会受影响。

### 4.6 SKU 排序

当前排序由 `DefaultSkuPriorityStrategy` 处理：

续作 SKU 使用主比较器：

1. 交期锁定优先。
2. 延误天数越小越优先。
3. 未来结构全收尾优先。
4. 供应链优先级待排量：高优、周期、中优、常规。
5. 物料编码稳定排序。

新增 SKU 使用专用比较器：

1. 先按施工阶段分组：试制、量试、小批量/正规。
2. 试制/量试组内按延误、目标量、物料编码。
3. 正规组内按定点机台、交期、延误、结构全收尾、供应链待排量。

评价：

- 排序层级清楚，且有优先级跟踪日志，便于解释真实批次顺序。
- “试制/量试不套完整供应链排序”是合理的局部规则，避免被正规 SKU 的供应链量级压制。
- 风险是排序对最终结果影响极大，而局部搜索尚未强力改写排序结果。后续业务变更应优先补排序日志验证，不要直接改主循环。

### 4.7 机台排序

当前机台匹配由 `DefaultMachineMatchStrategy` 处理：

硬过滤：

- 定点机台不可作业。
- 机台状态启用。
- 寸口、模套、特殊材料、模具占用。
- 长时间停机在有其他候选时后置或过滤。
- 单控/普通机台类型规则。

候选排序：

1. 限制作业定点机台优先。
2. 单控/普通机台按 SKU 类型优先。
3. 其他 SKU 占用情况。
4. 最早可生产班次。
5. 连续可生产班次。
6. 总可生产班次。
7. 尾部零散产能规避。
8. 普通机台优先，特殊能力少的机台优先。
9. 收尾时间、规格匹配、寸口距离、胶囊亲和、胎胚共用、机台顺序、机台号。

评价：

- 机台匹配区分“硬过滤”和“排序”，这是正确架构。
- 特殊材料、定点机台、单控机台都已有链路。
- `selectBestMachine` 只是返回排序后的第一个未排除候选，因此排序权重就是实际算法权重。若要优化算法，不应绕过排序，而应让评分、日志和局部搜索统一。

### 4.8 结果校验与持久化

S4.6 当前做了以下关键动作：

- 结果必填字段校验。
- 换模结果 `mouldCode` 校验。
- 生成换模计划并校验换模班次上限、手工周日喷砂阈值。
- 补工单号和排程序号。
- 添加汇总日志和日计划汇总日志。
- 应用硫化示方历史保护。
- 事务内原子替换目标日旧结果、未排结果、换模计划、过程日志。
- 发布排程完成事件。

优点：

- `replaceScheduleAtomically` 内有二次已发布校验，事务边界合理。
- 删除旧数据和插入新数据在一个事务内，降低目标日数据半更新风险。
- 保存前统一计算多机台班次收尾标记，有利于结果表字段一致。

潜在问题：

1. `validateProductionQuantityPolicy` 和 `validateGreenTireChangeoverShift` 当前保持历史关闭状态。对于多机台补满、同胎胚换模等复杂场景，这是可以理解的，但意味着部分数量强校验没有真正启用。
2. 工单号、换模工单号仍使用 `SimpleDateFormat`，与项目规范冲突。
3. 事件发布发生在持久化之后，但 Listener 异常不会影响主响应。需要确认精度计划回填和 MES 通知是否允许弱一致。

## 5. 排程算法层面分析

### 5.1 算法整体机制

当前算法不是全局最优化求解，而是工程型启发式排程：

```text
数据初始化
  -> SKU 目标量归集
  -> SKU 排序
  -> 续作优先贪心排产
  -> 换活字块衔接
  -> 续作降模修正
  -> 新增按排序逐个选机排产
  -> 多机台动态拆量
  -> 账本/库存/结果后置修正
  -> 保存前校验
```

其核心优势是可解释、可日志追踪、容易按业务规则局部修正。核心劣势是顺序敏感，前面的贪心选择会影响后续 SKU 的机台和窗口。

### 5.2 续作算法评价

续作算法的核心目标不是全局产能最大化，而是：

1. 保持在机连续性。
2. 优先收尾。
3. 必要时降模释放机台。
4. 剩余未满足量转新增补偿。

准确性评价：

- 对 MES 在机、滚动继承、同 SKU 多机台、dayN 账本都有考虑，业务匹配度较高。
- 对降模有 dayN 和 lookAheadDays 控制，不是简单按机台数砍。

效率评价：

- 主要是列表遍历和结果分组，复杂度可控。
- 大部分成本在多次账本同步、同 SKU 分组、机台状态重建。

主要风险：

- 同一 SKU 副本共享账本的对象关系复杂。
- 降模后补偿 SKU 回流 S4.5，若 sourceSku 或 dailyQuota 同步不完整，会导致新增阶段重复排或少排。

### 5.3 换活字块算法评价

换活字块算法的本质是“收尾机台优先的局部衔接搜索”：

- 先按机台收尾时间排序。
- 再在新增 SKU 中找同胎胚、同模具、同花纹等候选。
- 若命中定点物料，优先评估是否应让渡给新增换模链路。
- 每轮只落一条结果，然后重新按机台状态排序。

准确性评价：

- 能避免一次性按旧排序连续落多条结果。
- 能和定点机台、新增换模主链协调。

效率评价：

- 候选 SKU 和候选机台多时，每轮重新筛选会有一定开销，但相比新增主循环不是最大瓶颈。

主要风险：

- 换活字块候选规则散落在长类中，后续新增匹配层级时容易影响定点预留和 S4.5 回流。
- `scheduleType=03 + isTypeBlock=1 + isChangeMould=1` 三个字段的语义必须在下游统一，否则报表和 MES 可能误读。

### 5.4 新增换模算法评价

新增算法是当前系统最复杂的部分，主要依赖：

- SKU 排序决定处理顺序。
- 机台匹配排序决定首选机台。
- 换模均衡和首检均衡决定时间窗口。
- 日计划账本决定排多少。
- 多机台动态拆量决定是否继续扩机台。

准确性评价：

- 对试制、量试、小批量、正规、定点、特殊物料、清洗、保养、开停产等维度覆盖较完整。
- 正式新增非收尾满班补齐符合机台生产连续性，但会带来超日计划和超库存解释成本。

效率评价：

- 单 SKU 会多次计算候选机台、换模窗口、首检、班次产能。
- 类内存在大量回滚和重试分支，运行时复杂度主要由“SKU 数 x 候选机台数 x 班次数 x 重试次数”决定。
- 数据初始化已经并行化，后续性能瓶颈更可能在 S4.5 主循环和 S4.6 汇总校验。

主要风险：

- 贪心顺序导致局部最优，不保证全局产能或交期最优。
- 局部搜索目前有时间预算和 DFS 剪枝，但如果只用于评估而不改写最终候选，对结果提升有限。
- 换模/首检预占失败回滚依赖代码完整性，缺少专项回归容易污染全局资源计数。

### 5.5 局部搜索算法评价

`LocalSearchMachineAllocatorStrategy` 使用 DFS + 剪枝：

- 输入当前 SKU 和窗口 SKU 列表。
- 在时间预算内搜索候选机台组合。
- 以可行数和惩罚分更新最佳首台机台。
- 预占换模、首检和产能后回滚。

优点：

- 有时间预算，避免拖垮主流程。
- 有剪枝，避免完全枚举。
- 评估维度贴近真实换模和首检资源。

不足：

- 当前返回的是“首台机台”，不是完整排程方案。
- 如果 S4.5 主流程没有强制采纳该机台，局部搜索只能提供日志或弱建议。
- 惩罚函数和业务目标还不够体系化，暂未看到明确将发货、延误、结构收尾、定点、特殊材料等多目标统一评分。

建议：

1. 短期先明确局部搜索是否参与最终决策，避免“打开开关但结果无变化”的误判。
2. 若参与决策，返回对象不应只是 `MachineScheduleDTO`，而应包含机台、换模开始、首检、可排班次、预计排产量、评分明细。
3. 局部搜索只在候选规模小、冲突明显的场景启用，例如结构收尾、定点机台冲突、同胎胚多 SKU 竞争。

### 5.6 容量评估一致性风险

`TargetScheduleQtyResolver` 会基于候选机台评估窗口容量和结构收尾能力。当前需要重点关注一个一致性问题：

- 结果分配阶段会扣停机、清洗、保养、开停产等窗口。
- 目标量和结构收尾评估阶段部分计算未完全等价于结果分配阶段，例如容量评估中存在未传入真实清洗窗口的情况。

如果目标量评估高于实际可排产能，会导致：

- SKU 被标记为可收尾，但实际班次分配无法收尾。
- 结构全收尾排序提前，但后续结果不满足。
- 新增/续作目标量上调后再被结果阶段回裁，日志解释成本增加。

建议优先做“容量评估函数与班次分配函数一致性审计”，不急于改算法。

## 6. 具体架构问题与优化建议

### 6.1 P0：先补一致性校验和文档契约

不建议先拆大类。第一阶段应先降低误改风险。

建议任务：

1. 输出并维护当前主链路契约：S4.1-S4.6 每步输入、输出、必须维护的 context 字段。
2. 明确 `@DistributedLock` 是当前唯一入口锁；若 `ScheduleExecutionGuard` 不再使用，标记废弃或迁移。
3. 为 `scheduleResultSourceSkuMap` 增加保存前强校验：所有有计划量结果必须能追溯 sourceSku，除非是明确豁免的历史继承结果。
4. 为 `dailyPlanQuotaMap` 增加消费汇总日志：初始、续作消费、换活字块消费、新增消费、满班补齐超排、最终剩余。
5. 恢复或重写当前关闭的数量校验，但必须先用真实批次验证同胎胚换模和多机台补满，不可直接打开旧校验。

### 6.2 P1：小步拆分超大策略类

拆分原则：

- 不改变 S4.1-S4.6 主流程。
- 不改变现有表结构和 Mapper。
- 先拆私有方法和小型协作类，不引入新框架。
- 每次只拆一个职责，并用真实批次验证结果一致。

建议拆分顺序：

1. `NewSpecProductionStrategy`
   - 候选机台选择。
   - 换模/首检时间分配。
   - 单机台计划量决策。
   - 班次分配与日计划账本消费。
   - 结果构建与机台状态同步。
2. `ContinuousProductionStrategy`
   - 续作目标量规则。
   - 续作降模。
   - 续作补偿 SKU 回流。
   - 同 SKU 多机台尾量归集。
3. `TypeBlockProductionStrategy`
   - 换活字块候选 SKU 筛选。
   - 定点物料预留/让渡。
   - 换活字块结果构建。
   - 日计划账本回裁。
4. `ResultValidationHandler`
   - 结果字段校验。
   - 数量策略校验。
   - 换模计划生成。
   - 工单号和顺序号。
   - 汇总日志。

### 6.3 P1：统一日计划账本消费策略

当前续作、新增、换活字块都存在类似逻辑：

- 按业务日消费 `dailyPlanQuotaMap`。
- 收尾/严格目标截断。
- 非严格正式 SKU 允许满班补齐。
- 记录 `shiftFillOverQty`。

建议抽取为统一工具或小服务，例如：

```text
SkuDailyQuotaConsumptionService
  -> consumeForContinuous()
  -> consumeForNewSpec()
  -> consumeForTypeBlock()
  -> buildConsumptionSummary()
```

注意：不要一开始强行合并所有规则。可以先统一日志和结果对象，再逐步合并重复逻辑。

### 6.4 P1：统一容量评估与实际班次分配口径

建议建立两个对照用例：

1. 同一 SKU、同一机台、同一窗口下，目标评估容量与实际 `distributeToShifts` 可排量一致。
2. 存在清洗、保养、停机、晚班不可换模时，目标评估不高估实际可排量。

如果发现不一致，优先让目标评估复用现有窗口损失计算工具，而不是新增一套近似公式。

### 6.5 P2：让局部搜索成为可解释决策

当前局部搜索建议按三步演进：

1. 只记录：保留当前逻辑，但在日志中明确“局部搜索建议机台”和“最终采纳机台”是否一致。
2. 局部采纳：仅在建议机台仍在候选列表且不违反定点/试制/特殊材料硬约束时采纳。
3. 方案采纳：返回包含换模、首检、班次产能、评分明细的方案对象，主循环直接使用该方案，避免重复计算。

评分维度建议：

- 可行 SKU 数。
- 总未排惩罚。
- 发货锁定和延误惩罚。
- 结构全收尾奖励。
- 换模/首检资源消耗。
- 特殊机台占用惩罚。
- 定点机台违约惩罚。

### 6.6 P2：参数严格校验

建议新增配置项：

```text
ENABLE_STRICT_LH_PARAM_VALIDATION
```

行为：

- 关闭：保持当前 `warn + 默认值`。
- 开启：核心参数解析失败、低于最小值或互斥配置冲突时，S4.2 校验失败。

核心参数至少包括：

- 排程天数。
- 满产/需求模式。
- 换模限制。
- 局部搜索深度和时间预算。
- 欠产前瞻天数。
- 开停产比例。
- 清洗、保养、首检相关参数。

### 6.7 P2：观察者一致性分级

建议把 Listener 分为两类：

| 类型 | 示例 | 异常策略 |
| --- | --- | --- |
| 强一致后置动作 | 若未来要求排程成功必须完成某回填 | 失败应抛出并影响响应，或进入补偿任务 |
| 弱一致通知 | MES 通知日志、非关键提醒 | 失败只记录日志，可重试 |

当前 `PrecisionPlanScheduleDateFillListener` 是否允许弱一致，需要业务确认。

## 7. 业务逻辑修正建议

### 7.1 月计划余量口径

当前口径：`月计划总量 - 已完成量`。

建议：

- 若月余量表已废弃，仅保留当前口径，并在文档中明确。
- 若月余量表仍承载人工调整，则需要把人工调整量纳入 `ScheduleAdjustHandler`，但不能简单回退为旧兜底，应明确优先级：月计划原始量、完成量、人工调整、滚动继承、欠超产。

### 7.2 胎胚库存硬约束

当前正式续作/新增满排场景可能保留满班补齐结果，不一定被胎胚库存裁剪。

建议先让业务确认两种模式：

1. 胎胚库存硬约束：所有 scheduleType 都不得超过可用库存，超出必须未排。
2. 胎胚库存软约束：正式非收尾允许按机台满班补齐，超库存作为生产准备需求展示。

确认前不建议改代码。

### 7.3 换活字块字段语义

建议统一说明：

- `scheduleType=03` 表示换活字块排产。
- `isTypeBlock=1` 表示换活字块。
- `isChangeMould=1` 是为了生成下游切换计划，不等同普通新增换模。

下游 SQL、报表、MES 接口必须同时看 `scheduleType` 和 `isTypeBlock`。

### 7.4 试制/量试/小批量规则一致性

当前规则散落在：

- `ScheduleAdjustHandler`：施工阶段识别和目标量。
- `DefaultSkuPriorityStrategy`：新增排序分组。
- `DefaultMachineMatchStrategy`：单控/普通机台候选。
- `NewSpecProductionStrategy`：试制跳过、严格目标、普通机台放行。
- `DefaultTrialProductionStrategy`：试制专项规则。

建议建立一份“施工阶段规则表”，并由一个小工具类提供统一判断，避免各处判断条件漂移。

### 7.5 结果数量校验

当前保存前数量策略校验被注释关闭。

建议：

1. 先用真实批次找出历史关闭原因。
2. 将校验拆成三类：
   - 必须阻断：严格目标量超排、结果无来源 SKU。
   - 允许但需日志：正式 SKU 满班补齐超排。
   - 只统计不阻断：库存软约束超出。
3. 逐类打开，不要一次性恢复旧校验。

## 8. 性能瓶颈分析

### 8.1 当前主要性能来源

1. S4.2 并行加载大量基础数据，主要受数据库查询和线程池影响。
2. S4.3 遍历月计划并构建 SKU DTO，受月计划 SKU 数量影响。
3. S4.4 续作降模、多机台分组、账本同步。
4. S4.5 新增主循环，受 SKU 数、候选机台数、班次数、换模/首检重试影响。
5. S4.6 汇总日志、换模计划生成、批量落库。

### 8.2 优化建议

1. 对 S4.5 输出分段耗时日志：候选匹配、局部搜索、换模首检、班次分配、账本消费、结果构建。
2. 对 `matchMachines` 增加批次级缓存：同 SKU 在同阶段重复评估时，缓存硬过滤结果；注意机台状态变化后排序仍需重算。
3. 对结构收尾能力评估增加缓存，缓存 key 包含结构、机台状态版本、窗口日期。
4. 对过程日志和优先级日志设置批次级开关，避免生产常态下过量字符串拼接。
5. 局部搜索严格控制窗口规模，不要在所有新增 SKU 上默认深搜。

## 9. 风险点与回滚建议

### 9.1 高风险改动点

| 改动点 | 风险 |
| --- | --- |
| SKU 排序 comparator | 直接改变所有 SKU 上机顺序 |
| 机台排序 comparator | 直接改变落机结果、换模数量和收尾时间 |
| 日计划账本消费 | 影响计划量、欠产、超排、未排 |
| sourceSku 映射 | 影响库存裁剪、班次收尾、校验分组 |
| 续作降模 | 影响多机台保留、补偿 SKU、后续新增 |
| 换活字块字段 | 影响换模计划和下游接口识别 |
| 结果持久化 | 影响目标日旧数据覆盖和发布保护 |

### 9.2 回滚建议

1. 架构拆分类改动必须保持外部方法签名不变，便于 git 回滚。
2. 排序规则改动必须保留旧排序日志对比，可通过参数开关切换。
3. 局部搜索采纳必须有开关，默认可以先只记录不采纳。
4. 数量校验从“只日志”到“阻断”分阶段上线。
5. 持久化逻辑不建议与算法改动同批提交。

## 10. 验证建议

### 10.1 静态验证

1. `mvn -pl aps-lh -am -DskipTests compile`
2. 检查新增/修改类是否仍符合 Java 8。
3. 检查是否引入新的 XML、Mapper 或无业务依据的分支。

### 10.2 单元/局部验证

建议优先补以下可控测试：

1. SKU 排序：试制、量试、小批量、正规、交期锁定、延期、结构全收尾。
2. 机台排序：定点、单控、特殊材料、停机、尾部零散产能。
3. 日计划账本消费：续作、新增、换活字块共享同一账本。
4. 续作降模：同 SKU 多机台、dayN 不同、lookAheadDays 不同。
5. 换活字块：收尾机台、定点预留、回流 S4.5。
6. 结果校验：缺字段、严格目标超排、正式 SKU 满班补齐。

### 10.3 真实批次验证

真实验证应至少覆盖：

1. 纯续作，不换模。
2. 续作收尾后换活字块。
3. 续作多机台降模。
4. 新增正规 SKU 多机台分配。
5. 试制 SKU 单控机台。
6. 定点机台与普通候选冲突。
7. 特殊材料硬匹配。
8. 存在清洗、保养、停机窗口。
9. 满产模式与需求模式各一批。
10. 非强制重排滚动继承。

真实批次验收必须联查：

- `T_LH_SCHEDULE_RESULT`
- `T_LH_UNSCHEDULED_RESULT`
- `T_LH_MOULD_CHANGE_PLAN`
- `T_LH_SCHEDULE_PROCESS_LOG`
- 关键 SKU 的日计划账本汇总日志
- 关键机台的班次计划量和收尾标记

## 11. 分阶段实施方案

### 阶段一：只加校验和文档，不改算法

目标：降低误改风险。

任务：

1. 更新项目排程主链路文档，纠正 `ScheduleExecutionGuard` 未接入的事实。
2. 增加批次级配置快照日志。
3. 增加 sourceSku 完整性校验日志。
4. 增加日计划账本消费汇总日志。
5. 梳理关闭的数量校验原因。

预期影响：不改变排程结果。

### 阶段二：小步拆分超大类

目标：提升维护性，不改变业务结果。

任务：

1. 从 `NewSpecProductionStrategy` 抽取候选选择和班次分配私有协作方法。
2. 从 `ContinuousProductionStrategy` 抽取降模与补偿 SKU 逻辑。
3. 从 `TypeBlockProductionStrategy` 抽取候选筛选与账本回裁。
4. 从 `ResultValidationHandler` 抽取换模计划生成和工单号生成。

预期影响：理论上不改变排程结果；必须做真实批次 diff。

### 阶段三：算法增强

目标：减少贪心顺序带来的局部最优。

任务：

1. 明确局部搜索采纳开关。
2. 局部搜索输出评分明细。
3. 在定点机台冲突、结构收尾冲突、特殊材料冲突场景优先试点。
4. 建立“采纳前后结果对比报告”：排产量、换模数、未排数、收尾数、重点 SKU 落机。

预期影响：可能改变排程结果，必须走真实批次验证。

---

## 12. 最终建议

短期不要做大范围重构，也不要引入外部优化器。当前系统最需要的是把已经存在的复杂业务规则“契约化、校验化、日志化”，再按职责切小超大策略类。

推荐优先级：

1. 先修正文档和注释中不符合当前代码的入口锁描述。
2. 补 sourceSku、日计划账本、容量评估一致性校验。
3. 以不改变结果为目标拆分 `NewSpecProductionStrategy` 和 `ContinuousProductionStrategy`。
4. 明确胎胚库存硬约束/软约束口径。
5. 让局部搜索从“评估能力”逐步变成“可解释、可开关、可回滚的决策能力”。

这样改造风险最小，也最符合当前硫化排程系统频繁业务变更的现实。
